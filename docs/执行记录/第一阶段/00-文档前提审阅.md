## 2. 各文件核心结论摘要

### 2.1 产品需求

核心结论：

1. 第一阶段目标是交付“可导入文档、可稳定阅读、可按段落查看译文、可高亮与打标签、可保存进度、可离线恢复”的阅读 MVP。
2. 段落是第一阶段的主阅读单元，句子是段落内部的辅助单元；句子不能脱离段落结构独立主导主链。
3. 文本标准化、稳定段落生成、稳定句子切分是后续翻译、高亮、标签、恢复、缓存的共同前置条件。
4. 段落翻译是主翻译链；句子级翻译不是第一阶段主交付，但句子级辅助单元与句子级高亮属于第一阶段必须覆盖的能力。
5. 高亮必须绑定稳定结构锚点与文本快照，不能只靠显示顺序或纯字符偏移。
6. 阅读进度恢复不能只存滚动位置，必须优先基于文档 ID、阅读模式、稳定段落锚点与上下文定位。
7. 第一阶段离线能力必须覆盖已缓存内容的稳定可读，未缓存内容需要清晰提示，不要求整本书无条件全离线。
8. 只承诺文本型 PDF；OCR、扫描件识别、多格式导入、账号云同步、复杂协作都不在第一阶段范围内。

### 2.2 Orient

核心结论：

1. 当前仓库现实是 docs-only，尚无工程代码、依赖清单、测试框架与运行脚本。
2. 第一阶段必须先完成前提收敛，再启动技术实现，不允许把不存在的代码前提当成既有现实。
3. 方向上冻结为：Web/PWA、本地优先、文本型 PDF 优先、段落翻译主链、阅读恢复与离线为核心能力。
4. 第二阶段及以后再考虑 OCR、账号体系、跨端云同步、复杂格式与推荐类能力。

### 2.3 Design

核心结论：

1. 技术基线冻结为 `Vue 3 + TypeScript + Vite + Pinia + Vue Router`。
2. 核心实现分层包括：解析层、标准化层、领域模型层、状态层、存储层、缓存与离线层、阅读 UI 层、翻译 Provider 抽象层。
3. 关键数据对象包括：`Document`、`Paragraph`、`Sentence`、`Translation`、`Highlight`、`Tag`、`ReaderSession`、`CacheRecord`。
4. 主状态与派生状态必须分离，避免把解析结果、渲染状态、恢复状态、异步任务状态混在一起。
5. 通过 Gate A-E 控制阶段收口：基础运行、阅读主链、翻译与高亮、恢复与离线、测试与文档闭环。

### 2.4 Plan

核心结论：

1. 执行顺序总体正确：先工程基线，再解析与阅读，再翻译，再高亮，再恢复，再离线，再测试验收。
2. 任务树适合落地，但仍是“开发草案”，没有完全展开为用户要求的“每阶段目标/输入/输出/风险/DoD/测试点/记录文件”格式。
3. `Plan.md` 可作为任务来源，但需要重新编排为更适合顺序执行的 Phase 0-7 体系。

## 3. 第一阶段产品目标

第一阶段产品目标如下：

1. 用户可导入文本型 PDF，并在文档库中管理与切换文档。
2. 系统可基于稳定规则生成段落主单元与句子辅助单元。
3. 阅读页可在原文模式、双语模式等第一阶段定义的模式之间稳定切换。
4. 用户可按段落触发译文加载，并保证译文与原文稳定绑定。
5. 用户可对段落内句子或文本片段进行高亮、打标签、查看与跳转。
6. 用户关闭页面后重新进入，仍能恢复到正确文档、正确模式、正确内容附近。
7. 已缓存内容在离线状态下仍可阅读与恢复；未缓存内容需明示不可用边界。

## 4. 第一阶段必须完成的能力

必须完成：

1. 文本型 PDF 导入、解析、删除、文档列表管理。
2. 文本标准化。
3. 段落主单元生成。
4. 句子辅助单元生成与稳定锚点。
5. 阅读主视图与基础模式切换。
6. 段落翻译主链、翻译缓存、翻译失败反馈。
7. 句子级高亮与段落级上下文恢复。
8. 标签绑定、筛选、跳转。
9. 阅读进度保存与恢复。
10. 已读缓存、必要离线能力、离线提示。
11. 主状态/派生状态边界控制、错误状态反馈、验收与测试记录。

## 5. 第一阶段明确不做的能力

明确不做：

1. OCR 与扫描件识别。
2. Word、EPUB、网页剪藏等多格式导入。
3. 账号体系、云同步、多人协作。
4. 复杂推荐、社交、分享链路。
5. 句子级独立翻译链作为主体验。
6. 面向第二阶段的跨设备同步与在线资源中心。

## 6. 核心术语定义

| 术语 | 定义 | 第一阶段约束 |
| --- | --- | --- |
| 文档 | 用户导入并进入阅读流程的 PDF 实体 | 以本地文档为主，拥有稳定 `documentId` |
| 段落 | 第一阶段主阅读单元 | 翻译、恢复、缓存主链均以段落为主 |
| 句子 | 段落内部辅助单元 | 用于细粒度定位、高亮与局部上下文，不主导主链 |
| 文本标准化 | 清洗解析噪声并收敛文本边界的步骤 | 必须在段落/句子生成前完成 |
| 稳定锚点 | 用于跨渲染、跨缓存、跨恢复定位内容的结构标识 | 不允许只靠显示顺序 |
| 翻译绑定 | 原段落与目标译文之间的可追溯关联 | 以段落锚点绑定为主 |
| 高亮 | 用户对文本片段的标记行为 | 必须同时记录结构锚点与文本快照 |
| 标签 | 附着于高亮的分类信息 | 第一阶段不支持脱离高亮独立存在 |
| 阅读会话 | 用户在某文档上的阅读上下文快照 | 以文档、模式、锚点为核心 |
| 缓存 | 为已读内容与必要元信息提供离线可读能力 | 优先缓存已读，不承诺全量强制预取 |

## 7. 默认假设与边界

默认假设：

1. 输入文档以文本型 PDF 为主，解析后可获得可用文本层。
2. 文本标准化规则对同一版本文档是稳定且可重复的。
3. 第一阶段以单机本地数据为准，不处理跨端冲突合并。
4. 翻译服务通过 Provider 抽象接入，允许后续替换实现。
5. IndexedDB 类本地持久化能力可用；离线能力以 PWA + 本地缓存为基础。

边界约束：

1. 滚动位置仅作为辅助恢复信息，不能成为唯一恢复依据。
2. 翻译缓存、阅读缓存、高亮恢复都必须建立在同一套稳定段落/句子锚点之上。
3. 显示文本边界、翻译文本边界、缓存文本边界必须保持一致，禁止各自重新切分。

## 8. 模块依赖关系

依赖顺序如下：

1. 文档导入与解析 -> 文本标准化 -> 段落生成 -> 句子辅助单元生成。
2. 段落/句子稳定结构 -> 阅读视图渲染 -> 翻译绑定 -> 高亮与标签。
3. 稳定结构 + 阅读状态模型 -> 进度保存与恢复。
4. 稳定结构 + 翻译/高亮/进度数据 -> 缓存与离线恢复。
5. 所有主链完成后 -> 错误状态联调、回归测试、验收与清理。

## 9. 已知风险与历史坑点

高风险问题：

1. 原文与译文错位。
2. 句子切分不稳定，导致高亮与恢复锚点漂移。
3. 高亮恢复时因文本重新切分或标准化变动而失效。
4. 标签跳转依赖错误锚点，导致跳转位置错误。
5. 仅记录滚动位置导致重新打开后落点偏移。
6. 离线状态下因缓存粒度不一致而出现空白或半恢复状态。
7. 同一内容被不同链路重复切分、重复翻译、重复缓存。
8. 主状态与派生状态混杂，导致界面状态与持久化状态互相污染。

## 10. 文档冲突点、模糊点与统一解释原则

### 10.1 冲突与模糊点

1. `产品需求.md` 明确要求句子辅助单元与句子级高亮属于第一阶段能力；`Orient.md`、`Design.md` 部分表述更偏向“句子级翻译是可选增强”，容易被误读为句子相关能力整体可后置。
2. `产品需求.md` 强调恢复不能依赖纯滚动位置；`Design.md` 中的 `scrollOffset` 字段如果没有约束，容易被误用为主恢复依据。
3. `Plan.md` 的阶段拆分偏工程化，未完整暴露“专项记录、风险闭环、DoD、测试点、记录文件”这些执行控制要求。

### 10.2 统一解释原则

统一原则如下：

1. 业务边界冲突时，以 `产品需求.md` 为最高优先级。
2. 方向与非目标收口时，以 `Orient.md` 为准。
3. 技术实现与对象建模细节，以 `Design.md` 为准，但不得与 `产品需求.md` 冲突。
4. 开发顺序与任务树，以 `Plan.md` 为基础重组，但需服从本次重新编排后的 Phase 0-7 执行计划。
5. 句子级翻译可后置，但句子辅助单元、句子级定位、高亮恢复能力不得后置。
6. `scrollOffset` 仅允许作为“落点微调”，主恢复依据必须是文档 ID、阅读模式、段落锚点、句子锚点或上下文快照。

## 11. 第一阶段最终约束清单

最终冻结约束：

1. 先冻结边界与术语，再开始实现。
2. 先建立稳定阅读单元，再做翻译与高亮。
3. 先保证段落主链稳定，再引入句子辅助能力。
4. 先保证结构绑定正确，再做缓存与离线。
5. 先保证可恢复、可验证，再优化体验细节。
6. 全流程必须维持记录文件，可追溯每次决策、风险与测试。
