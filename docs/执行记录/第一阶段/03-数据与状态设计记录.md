# 第一阶段数据与状态设计记录

## 目标

第一阶段的数据与状态设计只服务以下闭环：

导入 -> 解析 -> 阅读 -> 原文/译文模式切换 -> 段落翻译 -> 段落/句子高亮 -> 标签 -> 恢复阅读 -> 已缓存内容离线可读

不在本阶段展开：

- 账号体系
- 云同步
- OCR
- TTS
- AI 总结/脑图
- 完整单词本或正式查词系统

## 当前已落地的数据对象

### 1. 文档元数据 `AppDocumentMeta`

用途：

- 支撑文档库列表
- 支撑最近阅读
- 支撑解析完成态、错误态、缓存态

核心字段：

- `id`
- `title`
- `createdAt`
- `lastOpenedAt`
- `lastReadAt`
- `pageCount`
- `paragraphCount`
- `coverSnippet`
- `parseCachedAt`
- `status`
- `lastError`

### 2. 解析结果 `ParsedDocumentCache`

用途：

- 作为阅读页连续正文的真实数据源
- 作为翻译、高亮、恢复、离线的结构基座

核心字段：

- `documentId`
- `title`
- `pageCount`
- `paragraphs`
- `cachedAt`
- `parserVersion`

### 3. 段落单元 `ParagraphUnit`

本阶段以段落作为主阅读单元。

核心字段：

- `id`
- `anchorKey`
- `order`
- `pageIndex`
- `text`
- `textHash`
- `sentences`
- `preview`

### 4. 句子单元 `SentenceUnit`

本阶段句子是辅助阅读单元，不承担主翻译绑定。

核心字段：

- `id`
- `text`
- `startIndex`
- `endIndex`

### 5. 翻译记录 `TranslationRecord`

本阶段翻译以段落为粒度持久化。

核心字段：

- `documentId`
- `paragraphId`
- `anchorKey`
- `sourceTextHash`
- `targetLanguage`
- `providerKey`
- `translatedText`
- `status`
- `updatedAt`
- `errorMessage`

### 6. 标注数据 `AnnotationBundle`

由高亮和标签组成。

高亮记录 `HighlightRecord`：

- `paragraph` 级高亮
- `sentence` 级高亮

标签记录 `TagRecord`：

- 标签绑定到高亮对象，不直接绑定裸文本

### 7. 阅读会话 `ReaderSession`

用途：

- 恢复当前文档
- 恢复当前段落
- 恢复阅读模式

核心字段：

- `documentId`
- `lastReadParagraphId`
- `mode`
- `lastUpdatedAt`

## 当前已落地的本地存储分层

使用 `localforage` 拆分为 6 个 store：

- `documents`
- `files`
- `parsed`
- `translations`
- `annotations`
- `sessions`

这样做的目的：

- 保持对象边界清晰
- 删除文档时可整体清理关联数据
- 恢复阅读时按层命中，不把所有状态混进一个巨大对象

## 当前 UI 状态与持久状态的分工

持久状态：

- 文档元数据
- 解析后的段落结构
- 翻译记录
- 高亮
- 标签
- 阅读会话

派生状态：

- 当前段落详情
- 当前模式按钮态
- 当前段落是否高亮
- 当前句子是否高亮
- 右侧详情面板内容
- 移动端底部详情弹层内容
- 全局状态条内容
- 轻 toast 提示

约束：

- UI 派生状态不能反向改写主结构
- 模式切换只改变展示方式，不改写段落结构
- 当前段落详情在 PC 和移动端展示位置不同，但读取同一套底层数据

## 2026-03-18 落地结论

本轮双端重构后，第一阶段实际运行所需的数据与状态对象已经连通：

- 导入后保存 PDF 文件与文档元数据
- 首次阅读时解析并缓存连续段落结构
- 阅读中实时保存阅读位置与模式
- 翻译、高亮、标签都能写回本地存储
- 重新进入文档时可从会话与缓存恢复阅读上下文

## 当前仍保留的问题

- PDF 解析仍属于文本型 PDF 优先策略，对复杂多栏版式的鲁棒性有限
- 句子高亮目前采用保守的“句子单元点击/选择”实现，不支持跨句自由选区
- 真正的生产级翻译质量仍依赖外部翻译接口，未配置接口时使用本地 glossary fallback
