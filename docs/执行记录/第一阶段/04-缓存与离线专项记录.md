# 第一阶段缓存与离线专项记录

## 目标

第一阶段的离线目标不是“整本书无条件全量离线”，而是：

- 已解析内容可缓存
- 已翻译内容可缓存
- 已缓存内容可离线继续阅读
- 高亮、标签、阅读位置可离线恢复
- 未缓存内容必须明确提示边界

## 当前已落地的缓存对象

- 文档元数据
- 原始 PDF 文件
- 解析后的段落结构
- 段落翻译记录
- 高亮与标签
- 阅读会话

## 当前已落地的缓存写入时机

### 1. 导入 PDF

- 写入 `documents`
- 写入 `files`

### 2. 首次打开文档

- 若未解析，则执行 PDF 解析
- 将解析结果写入 `parsed`
- 同步更新文档元数据中的 `pageCount`、`paragraphCount`、`parseCachedAt`

### 3. 阅读中触发翻译

- 将段落翻译写入 `translations`
- 成功态与失败态都保留，便于详情区展示与 Retry

### 4. 阅读中创建高亮与标签

- 将标注 bundle 写入 `annotations`

### 5. 阅读位置变化

- 将当前模式和当前段落写入 `sessions`

## 双端状态分工

### 全局状态

PC：

- 底部状态栏

移动端：

- 更轻的底部状态区

当前承载：

- `Cached`
- `Offline`
- `Saved locally`
- 当前页码/段落位置

### 当前段落状态

PC：

- 右侧详情面板

移动端：

- 底部详情弹层

当前承载：

- 当前段落翻译状态
- 翻译失败信息
- `Retry translation`
- 当前段落高亮与标签摘要

### 瞬时提示

- 使用 toast / 轻提示条
- 不把完成态塞回正文主区

## PWA 与离线壳

当前已使用 `vite-plugin-pwa` 生成 Service Worker。

作用：

- 缓存应用壳资源
- 让前端在离线时仍能启动
- 与 `localforage` 中的数据缓存形成配合

边界：

- 应用壳可离线启动，不代表所有文档内容都已离线就绪
- 文档内容是否可读，取决于该文档及相关段落是否已进入本地缓存链路

## 当前已知限制

- 未读到的段落不会被强制全量预取
- 未触发翻译的段落，离线时不会凭空出现译文
- 复杂 PDF 的图片层、扫描层、OCR 结果不在本阶段离线范围

## 2026-03-18 验证结论

从实现上看，第一阶段离线主链已成立：

- 有解析缓存时，原文连续阅读可恢复
- 有翻译缓存时，译文模式可命中已缓存内容
- 高亮、标签、会话都已本地持久化

但需要明确：

- 真正的断网浏览器人工验证尚未在当前 CLI 环境完成
- 当前“离线可读”结论来自状态链路实现、类型检查与生产构建验证

## 2026-03-18 移动端专项补充

本轮移动端专项对离线状态表达做了进一步收口：

1. 继续保留 `Cached`、`Offline`、`Saved locally` 的第一阶段状态分工。
2. `Cached` 仍以解析缓存为基础判断。
3. `Offline` 在移动端不再简单等同于“浏览器当前离线”，而是结合当前阅读模式与当前段落译文缓存情况派生。
4. 当处于译文模式且当前段落译文尚未缓存时，移动端会明确表达为“部分离线可读”或“当前译文未缓存”，避免把未缓存译文误显示为完全可离线。
