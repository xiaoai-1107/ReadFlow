## 1. 记录目的

本记录用于在 2026-03-19 基于第一阶段开发文档、执行记录与当前实现，对移动端阅读页做一次“阅读优先”的 UI 收口，并明确翻译问题的阶段归属。

## 3. 文档结论

### 3.1 第一阶段移动端必须保留的能力

1. 单列连续正文阅读流
2. `Original` / `Translation` 模式切换
3. 当前段落与邻近段落翻译触发
4. 翻译失败提示与 `Retry translation`
5. 段落级高亮
6. 句子级高亮
7. 标签绑定到高亮
8. `Structure` 结构导航
9. `My Study` 全局沉淀入口
10. `Details` 当前段落详情
11. 阅读位置与模式恢复
12. 已缓存内容离线可读
13. 明确但轻量的错误态、空态与状态提示

### 3.2 必须可达，但不要求固定大面积展示的入口

1. `Structure`
2. `My Study`
3. `Details`

文档一致要求移动端采用单列主阅读流，目录、学习沉淀和当前段落上下文更适合抽屉、底部弹层或轻浮层，而不是长期压在正文前方。

### 3.3 第一阶段对翻译能力的要求

第一阶段要求的是：

1. 段落翻译链路真实可运行
2. 翻译结果绑定稳定段落单元
3. 失败可见、可重试
4. 可缓存、可降级
5. 不阻塞原文阅读

当前执行记录同时明确：

1. 未接入真实 provider 时，`glossary fallback` 只是开发期降级链路
2. 不能把降级结果包装成“正式完成的真实翻译能力”

### 3.4 本轮允许调整的 UI 表达

1. 移动端顶部布局与模式切换承载方式
2. 正文容器 padding、字体、行高、段间距
3. `Structure / My Study / Details` 的入口形态与位置
4. 恢复提示与底部状态区的层级、厚度和常驻方式
5. 仅移动端生效的响应式样式和交互开关

### 3.5 本轮不能动的边界

1. 不新增第二阶段功能
2. 不改阅读主链逻辑
3. 不破坏高亮、标签、`My Study` 跳回正文
4. 不破坏模式切换与翻译触发机制
5. 不影响 PC 端现有稳定布局
6. 不把假翻译伪装成正式翻译完成态

## 4. 当前问题清单

### 4.1 顶部区域问题

1. 顶部控制区高度过高。
影响：首屏阅读效率、信息层级。
2. 返回、标题、模式切换、更多操作被放进同一块较厚的头部容器。
影响：首屏阅读效率、连续阅读体验。
3. 页面进入后正文出现偏晚，阅读启动成本高。
影响：首屏阅读效率、功能认知一致性。

### 4.2 正文排版问题

1. 移动端正文左右 padding 偏大。
影响：连续阅读体验。
2. 外层阅读卡片与内层正文同时收缩，有效阅读宽度不足。
影响：连续阅读体验、信息层级。
3. 单行字符数偏少，换行过频。
影响：连续阅读体验。
4. 留白看起来很多，但正文视觉上仍显拥挤。
影响：连续阅读体验、信息层级。

### 4.3 入口布局问题

1. `Structure / My Study / Details` 作为三枚大按钮直接放在正文前。
影响：首屏阅读效率、信息层级。
2. 入口虽然可见，但层级高于正文。
影响：信息层级、功能认知一致性。
3. `My Study` 在无高亮时会被交互拦截，导致“入口不可达”。
影响：入口可达性、功能认知一致性。

### 4.4 状态提示问题

1. 恢复提示、顶部区和正文共同争抢垂直空间。
影响：首屏阅读效率、连续阅读体验。
2. 底部状态区仍有一定存在感，容易和正文竞争注意力。
影响：连续阅读体验、信息层级。

### 4.5 翻译能力问题

1. UI 已暴露 `Translation` 模式。
影响：功能认知一致性。
2. 但当前默认链路仍可能落到 `glossary fallback`，不是正式生产级翻译。
影响：功能认知一致性。
3. 这属于第一阶段翻译能力未完全闭环，而不是单纯 UI 问题。
影响：功能认知一致性、阶段验收判断。

## 5. 收口方案

### 5.1 顶部布局

1. 移除移动端“大卡片式三排头部”倾向。
2. 改为更轻的两层顶部结构：
   - 第一层：返回、标题、更多操作
   - 第二层：更紧凑的 segmented control
3. 头部视觉减重，滚动时只保留轻工具栏语义，不再成为主视觉。

### 5.2 正文容器

1. 移除移动端正文的卡片化包裹感。
2. 下调正文左右内边距。
3. 缩小段落索引与正文之间的横向占用。
4. 微调正文字号、行高和段间距，让单列阅读更自然。

### 5.3 入口形态

本轮采用：

1. 单一悬浮入口 `Reader tools`
2. 点击后展开轻量底部 action sheet
3. 在 action sheet 中进入 `Structure`、`My Study`、`Details`

这样处理后可满足：

1. 入口始终可达
2. 平时不挡正文
3. 不再在首屏正文前堆按钮
4. 与底部状态区错层存在

### 5.4 恢复提示与状态区

1. 恢复提示改为更短时的轻提示，不再长期压在正文前。
2. 移动端底部状态区进一步变薄、变轻。
3. 保留 `Cache / Offline / Local` 状态语义，但压低视觉权重。

## 6. 翻译问题结论

结论 C：当前实现可作为开发阶段降级方案，但不能视为第一阶段正式完成态。

理由：

1. 文档要求第一阶段翻译链路真实存在、可失败重试、可缓存、可降级。
2. 当前版本已满足“链路存在且可运行”的要求。
3. 但执行记录已明确：未接入真实 provider 时，`glossary fallback` 只是降级方案。
4. 因此当前不能把它描述成“正式完成的真实翻译能力”，更不能把这个问题混成纯 UI 问题。
