# 第一阶段移动端功能梳理与落地记录

## 1. 记录目的

本记录用于在移动端专项开发开始前，先收敛第一阶段移动端的产品边界、UI 结构、当前实现现状与差距，再据此推进真实功能开发与测试。

本轮目标不是补一层响应式样式，而是确认移动端 Web / PWA 在第一阶段内应具备哪些真实能力，以及哪些问题会阻断“可实际使用”。

## 3. 第一阶段移动端必须功能清单

结合第一阶段产品需求、UI 设计需求、双端 UI 重构记录与测试计划，移动端在第一阶段必须支持以下真实能力：

1. PDF 导入与进入阅读
2. 单列连续正文阅读流
3. 原文模式
4. 译文模式
5. 当前段落与邻近段落优先翻译
6. 翻译失败提示与 `Retry translation`
7. 段落级高亮
8. 句子级高亮
9. 标签绑定到高亮
10. 当前段落详情底部弹层
11. 目录入口 / 抽屉
12. My Study 全局沉淀入口
13. 阅读位置与阅读模式恢复
14. 解析缓存、翻译缓存、高亮/标签/会话的本地持久化
15. 已缓存内容离线可读
16. 未缓存内容边界提示
17. 轻量错误状态与 toast 反馈

## 4. 第一阶段移动端 UI 结构

移动端必须采用以下结构，而不是把 PC 缩放到小屏上：

1. 顶部轻工具栏
2. 单列连续正文阅读流
3. `Original` / `Translation` 两个模式按钮
4. 选中句子后临时出现的底部操作条
5. 当前段落详情底部弹层
6. `Structure` 抽屉或等价入口
7. 更轻的底部状态区

对应的交互角色为：

- `Structure`：阅读结构导航
- `My Study`：当前文档的全局沉淀入口
- `Details`：当前段落局部摘要与状态承载
- 临时底部操作条：句子选中后的短时操作层

## 5. 移动端与 PC 的关系

第一阶段双端关系必须保持：

1. 核心数据一致
2. 状态逻辑一致
3. 交互语义一致
4. 展示结构不同

具体表现为：

- PC 用右侧详情面板承载当前段落上下文
- 移动端用底部详情弹层承载相同语义
- PC 用选区附近轻浮层承载句子操作
- 移动端用临时底部操作条承载同样的操作
- PC 有多区域布局
- 移动端必须是单列阅读流

## 6. 第一阶段边界

移动端本轮不得误做以下第二阶段能力：

1. 单词本
2. 查词系统正式版
3. 句子级翻译主链
4. AI 总结
5. 导出学习材料
6. 账号体系
7. 云同步
8. 复杂复习系统

允许保留最小扩展余量，但不在本轮正式展开。

## 7. 当前移动端实现现状

当前代码中，移动端已经不是纯 UI 壳，已具备以下真实实现基础：

1. `ReaderView.vue` 已通过 `isMobile` 进入移动端布局分支。
2. 移动端已有顶部工具栏、模式切换、正文区、底部状态区。
3. 移动端已有 `Structure` 抽屉。
4. 移动端已有当前段落详情底部弹层。
5. 句子选中后已有临时底部操作条。
6. `readerStore` 已连通真实解析、翻译、模式切换与恢复逻辑。
7. `highlightStore` 已连通真实高亮、标签与持久化逻辑。
8. `sessionStore` 已连通阅读模式与当前位置恢复。
9. `storage/index.ts` 已把文档、解析、翻译、标注、会话拆入本地持久化分层。

因此，本轮移动端专项不是从零开始，而是在现有真实链路上补齐移动端未闭合部分。

## 8. 当前移动端与需求差距

### 8.1 已有但需继续验证的能力

1. 原文模式切换已存在，但需验证正文是否始终只显示英文。
2. 译文模式切换已存在，但需验证是否始终是“一段原文 + 一段译文”的上下结构。
3. 当前可见段落触发翻译已存在，但需验证移动端详情弹层中的失败状态与重试链路是否完整。
4. 句子选中底部操作条已存在，但需验证是否严格遵守“默认不显示，取消选中后消失”。
5. 当前段落详情弹层已存在，但需要更明确地承担“段落位置 + 翻译状态 + 高亮摘要 + 标签摘要”的角色。

### 8.2 当前仍偏“近似实现”的点

1. 移动端正文仍复用桌面阅读节点结构，需要继续确认不存在桌面侧栏语义残留。
2. 移动端详情弹层中的信息层级仍偏桌面式，需要更像“当前段落底部详情”。
3. 底部状态区虽然已存在，但当前“Cached / Offline / Saved locally”的表达仍偏粗粒度。
4. 阅读恢复提示虽然已存在，但需要继续控制占屏与关闭后不打断阅读。

### 8.3 当前会阻断“移动端可实际使用”的问题

1. 移动端入口链路不够完整：阅读页顶部没有独立的移动端导入入口，用户更多依赖文档库进入。
2. 离线状态表达仍主要基于解析缓存命中，尚未把“译文是否已缓存”表达得足够清楚。
3. 需要确认移动端 `My Study`、`Details`、`Structure` 三个入口的职责没有再次重叠。
4. 需要做移动端专项测试，确认真实功能不是“看起来有，实际链路断裂”。

## 9. 本轮实现优先级

### 优先级 1：移动端主阅读与状态闭环

先确保以下链路在移动端是真闭环：

1. 单列连续阅读
2. 原文 / 译文模式切换
3. 当前段落底部详情
4. 翻译失败与重试
5. 轻量恢复提示
6. 全局底部状态区

### 优先级 2：移动端交互闭环

1. 句子选中后临时操作条
2. 段落高亮
3. 句子高亮
4. 标签绑定
5. 从详情层或 My Study 回到正文位置

### 优先级 3：移动端缓存与离线边界表达

1. 解析缓存命中
2. 翻译缓存命中
3. 离线时已缓存内容可读
4. 未缓存内容提示边界

## 10. 本轮实现范围

本轮移动端专项只做第一阶段必须闭合的能力：

1. 完善移动端阅读页主布局和移动端专用交互承载
2. 打通移动端翻译、高亮、标签、详情、恢复、缓存状态的真实链路
3. 补齐移动端测试与记录

本轮不做：

1. 第二阶段理解增强
2. 单词本 / 查词 / 导出
3. 云同步
4. 真机特有手势细节的大规模重设计

## 11. 等价实现说明

若个别设计细节在现有代码中无法完全一比一还原，可接受的第一阶段等价实现包括：

1. 用移动端底部详情弹层承载 PC 右侧详情面板的同等信息，而不是强做常驻侧栏。
2. 用临时固定底部操作条承载移动端句子选中操作，而不是依赖浏览器原生长按菜单。
3. 用本地 glossary fallback 作为未配置外部翻译接口时的开发降级翻译，而不是假成功态。

## 12. 当前仍需重点验证的遗留问题

1. 真机浏览器下的长按/点按体验仍需继续验证。
2. 复杂 PDF 的段落切分和移动端长文滚动手感仍需更多样本回归。
3. 离线状态的细粒度表达仍可进一步细化为“原文已缓存 / 译文已缓存”的更精确版本。

## 13. 当前结论

移动端第一阶段所需的主结构与核心数据链路已经存在，但仍需要一次围绕“真实可用而不是展示态”的专项收口。本轮开发应以本记录为依据，补齐移动端布局、状态、交互和测试闭环，同时不破坏 PC 已有主链。

## 14. 本轮已实现的能力

### 14.1 移动端当前段落详情底部弹层

已补齐为更明确的第一阶段详情结构：

1. 当前段落位置
2. 当前段落预览
3. 翻译状态
4. 当前段落高亮摘要
5. 当前段落标签摘要
6. 进入 `My Study` 的轻入口

### 14.2 移动端句子选中后的临时操作条

已继续收口为真实临时交互：

1. 只有选中句子时出现
2. 执行 `Highlight` 后自动退场
3. 执行 `Add tag` 并保存后自动退场
4. 执行 `Copy` 后自动退场

### 14.3 移动端离线状态表达

已将底部状态区与详情弹层中的离线表达改为更贴近真实缓存能力：

1. 原文缓存与译文缓存不再被混为一个“永远可离线”的假状态。
2. 当译文模式下当前段落译文尚未缓存时，会明确表达为部分可离线或当前译文未缓存。
3. 当前段落详情中会补充当前译文是否已保存到本地的提示。
