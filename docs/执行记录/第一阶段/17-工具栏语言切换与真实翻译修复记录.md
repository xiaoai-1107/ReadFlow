# 17-工具栏语言切换与真实翻译修复记录

## 1. 记录目的

本记录用于在 2026-03-20 对阅读页补齐两项第一阶段收口工作：

1. 增加阅读页工具栏界面语言切换能力
2. 修复译文模式对假翻译 / glossary fallback 的误判问题，明确真实翻译状态边界

本次工作不新增第二阶段能力，不改变第一阶段阅读主链，只对阅读页 UI 文案体系、翻译状态模型和前端展示逻辑做收口。

## 2. 当前翻译链路核对结论

### 2.1 改造前的真实情况

核对文件：

- `src/services/translation/index.ts`
- `src/stores/reader.ts`
- `src/pages/ReaderView.vue`
- `src/services/storage/index.ts`

改造前结论：

1. `Translation` 模式的数据来源由 `readerStore.ensureTranslation()` 触发
2. 若配置了 `VITE_READFLOW_TRANSLATION_ENDPOINT`，会请求外部 endpoint
3. 若未配置 endpoint，会走 glossary fallback
4. fallback 结果会被写入翻译缓存
5. 前端会把 fallback 结果当成正常译文完成态展示
6. 这会造成“看起来已有译文，实际并非真实翻译”的误导

### 2.2 本次改造后的结论

本次将翻译链路收敛为以下规则：

1. 只有真实 provider / endpoint 返回的数据，才允许进入正式译文成功态
2. 未配置真实翻译服务时，前端进入 `unavailable / provider_not_configured` 状态
3. glossary / fallback 不再被当成正式译文成功写回并展示
4. 已有旧 fallback 缓存会在读取时被归一化为 `unavailable`，不再伪装成已翻译完成
5. Retry 仅针对真实失败态显示，不再对“未配置 provider”显示误导性重试

## 4. 真实翻译改造说明

### 4.1 状态字段调整

在翻译记录中补齐并明确以下字段：

- `status: 'idle' | 'translating' | 'translated' | 'failed' | 'unavailable'`
- `source: 'provider' | 'fallback' | 'mock' | 'none'`
- `errorCode: 'provider_not_configured' | 'offline' | 'request_failed' | 'empty_result' | 'unknown' | null`

### 4.2 UI 判定规则

前端现在只将以下组合视为正式可展示译文：

- `status === 'translated'`
- `source === 'provider'`

其余情况统一按真实状态展示：

1. `translating`：显示真实翻译加载中
2. `failed`：显示失败原因与 retry
3. `unavailable` 或 `provider_not_configured`：明确提示当前未接入真实翻译服务
4. 离线且无真实缓存：提示不可离线获取，不伪装为已可读译文

### 4.3 服务层改造

`src/services/translation/index.ts` 本次收口后：

1. 未配置 endpoint 时直接抛出 `provider_not_configured`
2. 离线时抛出 `offline`
3. 请求失败时抛出 `request_failed`
4. 空结果时抛出 `empty_result`
5. 只有真实 provider 返回结果才返回 `source: 'provider'`

## 5. 状态字段与 UI 文案映射

| 场景 | 记录状态 | UI 表达 | 是否允许展示译文正文 |
| --- | --- | --- | --- |
| 真实翻译成功 | `translated + source=provider` | 展示译文正文，显示已缓存 provider | 是 |
| 翻译中 | `translating` | 显示加载提示 | 否 |
| 真实翻译失败 | `failed` | 显示失败原因与 Retry | 否 |
| 未配置真实服务 | `unavailable` / `provider_not_configured` | 显示未接入真实翻译服务 | 否 |
| 历史 fallback 缓存 | 读取后归一化为 `unavailable` | 不再展示为成功译文 | 否 |
