# 第一阶段双端 UI 重构与落地记录

## 目标

基于最新确认的 UI 稿，把当前项目从“可展示页面”推进到“可实际使用的软件界面与状态流”。

## PC 端实际改动

1. 重建顶部工具栏，统一承载返回、文档名、模式切换、搜索与更多操作
2. 重建左侧导航区，包含当前文档、Structure、My Study
3. 将主阅读区改为连续正文流
4. 原文模式只展示英文正文
5. 译文模式改为原文在上、译文在下的连续阅读结构
6. 建立右侧当前段落详情面板
7. 建立底部全局状态栏
8. 建立选中句子后的轻浮层

## 移动端实际改动

1. 重建轻量顶部工具栏
2. 将主阅读区改为单列连续正文流
3. 保留原文/译文双模式切换
4. 将目录改为抽屉入口
5. 将当前段落详情改为底部弹层
6. 将句子选中操作改为临时底部操作条
7. 将全局状态压缩为更轻的底部状态区
8. 将恢复阅读提示改为更轻的顶部提示

## 双端统一的交互逻辑

1. 原文模式定义一致
2. 译文模式定义一致
3. 当前段落详情结构一致
4. 选中句子操作规则一致
5. 全局状态 / 当前段落状态 / 瞬时提示的分工一致
6. 恢复阅读逻辑一致

## 这轮不是纯 UI 还原的地方

本轮同时落地了真实状态来源：

- 详情层读取真实翻译记录
- 状态条读取真实缓存与会话状态
- 高亮与标签读取真实标注数据
- 恢复提示来自真实会话恢复结果

## 因技术原因采用的等价替代

1. 当前“句子选中”实现偏向句子级选择，而不是任意文本 range 选区
2. 标签编辑使用右侧/底部的轻量输入区，而不是复杂弹窗工作流
3. 未配置真实翻译 endpoint 时，译文来自 glossary fallback，而不是外部翻译服务

## 当前仍需继续优化的点

1. 移动端原生长按与选区体验
2. 复杂 PDF 版式下的段落切分稳定性
3. 真机状态栏与弹层高度的细节打磨
4. 真实翻译服务接入后的文案与失败回退策略

## 2026-03-18 移动端专项补充

### 移动端本轮继续收口的点

1. 将底部详情弹层进一步收口为“当前段落详情”角色，而不是泛化信息弹层。
2. 将临时底部操作条进一步收口为“选中后出现、操作后退场”的移动端专用交互。
3. 将底部状态区对离线边界的表达进一步贴近真实缓存状态。

### 仍属于等价实现的点

1. 移动端句子选中仍以句子级点击/选择为主，不是浏览器原生自由文本 range。
2. 移动端离线状态仍以第一阶段缓存边界为准，不展开更复杂的分层缓存中心。
