# 第一阶段翻译与锚点绑定专项记录

## 目标

保证第一阶段段落翻译链路真实可用，并且不发生原文与译文错位。

## 主绑定原则

- 段落是翻译主单元
- 句子是辅助阅读单元
- 翻译不按屏幕顺序绑定
- 翻译不按 DOM 顺序绑定
- 翻译不按临时列表索引绑定

## 当前实际绑定键

翻译记录以以下信息绑定：

- `documentId`
- `paragraphId`
- `anchorKey`
- `sourceTextHash`
- `targetLanguage`

这样做的目的：

- 避免滚动后错绑
- 避免模式切换后错绑
- 避免重进页面后错绑
- 避免文本变化时继续复用旧译文

## 当前翻译触发策略

- 当前段落进入阅读焦点后优先触发
- 同时轻量预取相邻段落
- 若已有成功缓存，则直接命中缓存
- 若正在翻译，则不重复请求
- 若失败，可手动 Retry

## 当前翻译提供方式

### 优先方式

若存在 `VITE_READFLOW_TRANSLATION_ENDPOINT`：

- 调用外部 HTTP 翻译服务

### 降级方式

若未配置外部 endpoint：

- 使用本地 glossary fallback 生成可读的降级译文

说明：

- 这不是“假成功态”
- 它会真实返回降级翻译结果并持久化
- 但它不是生产级翻译质量

## UI 对应关系

原文模式：

- 正文只显示英文
- 当前段落译文不进入正文主流
- 右侧详情/底部详情只展示当前段落译文状态与内容

译文模式：

- 每段保持“原文在上，译文在下”
- 仍然是连续阅读流
- 不做左右双栏

## 失败处理

- 翻译失败不阻塞原文阅读
- 失败态明确展示在当前段落详情层
- 支持 `Retry translation`

## 2026-03-18 落地结论

翻译主链已经完成软件级连通：

- 阅读页会按当前段落与邻近段落触发翻译
- 结果与段落结构稳定绑定
- 成功、失败、重试、缓存都已经有真实状态
- 译文模式可消费同一条翻译缓存链路

## 当前遗留

- 若要达到正式上线质量，必须接入真实翻译服务 endpoint
- 当前 glossary fallback 适合作为开发环境与离线降级，不适合作为最终翻译质量标准
