## 1. Summary

- 一句话定义: ReadFlow MVP 是一个面向英文原版 PDF 的 Web/PWA 阅读辅助工具，第一阶段必须打通“导入、解析、阅读、翻译、标注、恢复、离线继续阅读”的完整闭环。
- 当前最重要结论:
  1. 本轮不是修一个 `documentStore`，而是为完整 MVP 建立上游冻结边界。
  2. 当前仓库是纯文档起点，后续 Design 和 Plan 必须显式覆盖工程脚手架、实现目录、测试和验收路径。
  3. 会话恢复与离线可读不是优化项，而是 MVP 主链能力。

  1. 不得把任务再次缩窄成“状态管理局部重构”。
  2. 不得假设存在仓库里并不存在的代码基础。
  3. 不得把离线、恢复、错误反馈降级成后置优化。

## 3. 任务背景与当前事实

### 3.1 产品背景

根据 `产品需求.md`，ReadFlow 的第一阶段目标不是做一个通用 PDF 工具，而是做一个帮助用户持续阅读英文原版 PDF、降低理解门槛并沉淀重点内容的阅读辅助工具。

产品主链明确为：

`导入 -> 解析 -> 阅读 -> 翻译 -> 标注 -> 恢复 -> 离线继续阅读`

### 3.2 当前仓库现实

1. 当前仓库尚未进入实现阶段，只有文档，没有任何应用代码。
2. 因为没有现有工程，后续 Design 必须补齐工程骨架、目录、测试、脚本、存储、调试与验收路径。
3. 因为没有现有实现，Plan 也不能再围绕“迁移旧 store”拆任务，而要围绕“搭建 MVP”拆任务。

### 3.3 上一版文档的主要偏差

1. 范围偏差: 把完整产品问题收窄成单一 `documentStore` 状态管理问题。
2. 前提偏差: 引用了仓库里不存在的 `src/store/*` 和“当前实现”。
3. 阶段偏差: `Orient` 中提前冻结了大量实现级 store 结构，职责越界到 `Design`。
4. 覆盖不足: 对导入、阅读壳层、PWA、测试、工程脚手架、错误反馈、验收路径覆盖不完整。
5. 执行不可用: `Plan` 任务直接依赖不存在的代码对象，不能真正驱动 0→1 落地。

### 3.4 当前阶段一句话定义

在纯文档起点上，为 ReadFlow MVP 冻结完整产品与工程落地边界，而不是继续围绕不存在的既有状态管理实现做局部设计。

## 4. 核心目标与成功标准

### 4.1 本轮核心目标

1. 冻结 ReadFlow MVP 第一阶段的真实交付对象。
2. 冻结必须优先打通的用户主链，而不是追求功能堆叠。
3. 冻结当前仓库的真实项目起点和工程范围。
4. 冻结后续 Design 必须展开的技术主题与不可回退的方向。
5. 冻结后续 Plan 必须承接的实施模块。

### 4.2 成功标准

1. 任何人阅读本文后，都能明确本项目当前要交付的是“ReadFlow MVP”，而不是某个内部状态模块。
2. 后续 `Design` 不再假设不存在的代码基础。
3. 后续 `Design` 必须完整覆盖导入、解析、阅读、翻译、标注、恢复、离线、错误反馈与工程基建。
4. 后续 `Plan` 可以直接据此拆成工程可执行任务，而不需要重新讨论产品边界。

### 4.3 失败判定

出现以下任一情况，视为 `Orient` 失败：

1. 后续文档再次把范围收窄成局部 store 或局部状态问题。
2. 后续文档默认仓库里已经有实现和目录结构。
3. 离线、恢复、错误反馈、验证路径没有被提升为主链能力。
4. 文档没有明确 `Out of Scope`，导致 MVP 无边界扩张。

## 5. 现实约束

### 5.1 产品约束

1. 第一阶段只做英文原版 PDF 阅读辅助闭环。
2. 第一阶段默认目标语言为中文。
3. 第一阶段必须支持“下次继续读”和“离线读已缓存内容”。
4. 第一阶段不引入账号、云同步、社交、支付、多人协作。

### 5.2 项目约束

1. 当前仓库从零开始，没有可继承代码。
2. 当前阶段首先要补齐工程骨架与基础设施，否则无法进入执行。
3. 文档需要为后续自动化执行提供明确路径，不能停留在概念层。

### 5.3 工程约束

1. MVP 必须以 Web/PWA 为第一交付形态。
2. MVP 必须是本地优先设计，至少本地保存文档、翻译、标注和会话数据。
3. PDF 首阶段默认只保证“文本型 PDF”的稳定体验，不承诺 OCR 扫描件质量。
4. 错误路径必须显式建模，不能依赖“失败时页面没崩就算可用”。

### 5.4 验收约束

1. 不能只以页面可打开为完成标准。
2. 必须能做完整主链演示。
3. 必须有本地验证路径，而不是只留概念方案。

## 6. In Scope / Out of Scope

### 6.1 In Scope

1. Web/PWA 端 ReadFlow MVP 工程脚手架。
2. PDF 导入、基础校验、文档列表、切换与删除。
3. PDF 文本解析、段落组织、稳定锚点与阅读展示。
4. 原文 / 译文 / 双语三种阅读模式。
5. 段落级翻译，句子级翻译作为可选增强位。
6. 高亮、标签、重点回看与跳转。
7. 阅读进度保存与恢复。
8. 已缓存内容的离线继续阅读。
9. 导入、解析、翻译、缓存、恢复等关键错误反馈。
10. 本地测试、验收脚本、文档回写与 Gate。

### 6.2 Out of Scope

1. 账号登录、用户体系、支付、商业化策略。
2. 云同步、多端实时协同、多人批注。
3. OCR 扫描件增强、复杂学术排版适配。
4. TTS、听书、录音、社交分享、社区能力。
5. AI 脑图、复习系统、单词本、复杂导出工作流。
6. 原生移动端 App。

## 7. 冻结决策（O-01 ~ O-12）

| ID | 决策 | 冻结值 |
|---|---|---|
| O-01 | 项目性质 | 当前项目是 docs-only 的 0→1 MVP 项目 |
| O-02 | 交付形态 | 第一阶段交付 Web/PWA，而不是原生多端 |
| O-03 | MVP 对象 | 面向完整 ReadFlow 阅读闭环，不再缩窄为局部状态管理问题 |
| O-04 | 主链优先级 | `导入 -> 解析 -> 阅读 -> 翻译 -> 标注 -> 恢复 -> 离线继续阅读` |
| O-05 | 核心阅读粒度 | 段落为主粒度，句子为可选增强粒度 |
| O-06 | 数据策略 | 本地优先，必须支持本地持久化会话、标注与已缓存内容 |
| O-07 | 离线策略 | 离线可读属于 MVP 主链能力，不可后置 |
| O-08 | 恢复策略 | 阅读会话恢复属于 MVP 主链能力，不可后置 |
| O-09 | 翻译策略 | 翻译必须通过可替换 Provider 接口接入，不把具体服务写死到产品逻辑 |
| O-10 | 文档边界 | 第一阶段以文本型 PDF 为主，不把 OCR 能力纳入 MVP 承诺 |
| O-11 | 错误策略 | 关键失败路径必须显式暴露、可重试、可降级 |
| O-12 | 文档流程 | 未经用户 Check，`Orient -> Design -> Plan -> M` 不能被视为正式批准链路 |

## 8. 对后续 Design 的强约束

后续 `Design` 必须覆盖并冻结以下内容：

1. 工程技术基线与目录落位，而不是继续引用不存在的现有实现。
2. 导入、解析、阅读、翻译、标注、恢复、离线、错误反馈的完整产品级模块划分。
3. 本地持久化策略、离线策略、缓存对象和会话恢复顺序。
4. PDF 解析与段落锚点建模方式。
5. 翻译 Provider 适配层、缓存策略与降级策略。
6. 高亮、标签、阅读会话之间的数据关系。
7. 调试、测试、验收脚本、Gate 和回滚纪律。

### 8.1 Design 不得回退的方向

1. 不得再次假设仓库已有 `src/store/*` 现实。
2. 不得只做“状态树设计”，忽略完整产品与工程主链。
3. 不得把测试、构建、PWA、离线、错误反馈留成“后面再说”。
4. 不得把会话恢复、离线可读降级成附属优化。
5. 不得通过模糊表述把关键选择拖到 `Plan` 或 `M` 阶段再决定。
