# 阅流 ReadFlow（产品需求）

> 目标文件: `产品需求.md`  
> 版本: v1.0-draft-02  
> 日期: 2026-03-16  
> 状态: Draft  
> 用途: 定义阅流 ReadFlow 作为一个全新项目的产品目标、目标用户、核心场景、功能范围、非功能要求与阶段验收标准，为后续 Orient / Design / Plan 提供上游依据。

---

## Summary

1. 阅流 ReadFlow 是一个面向英文原版 PDF 阅读场景的辅助理解工具，核心目标是降低原版阅读门槛，并帮助用户在阅读过程中完成翻译、标注、整理与持续复习。
2. 本项目从 0 开始重新定义，不继承任何旧项目的代码结构、状态设计、缓存策略或历史实现约束。
3. 第一阶段不追求“大而全”，而是先完成“可导入、可阅读、可翻译、可标注、可恢复”的最小可用闭环。
4. 产品的长期方向不是普通 PDF 阅读器，而是“阅读 + 理解 + 标注 + 提取 + 沉淀”的持续型阅读工具。
5. 第一阶段默认聚焦“文本可提取的英文 PDF”，优先保证阅读主链稳定，而不是追求覆盖所有 PDF 类型和复杂场景。

---

## 0. 文档定位

### 0.1 本文解决什么问题

本文档解决的是：  
如果把“英文原版 PDF 阅读工具”当成一个**全新产品**来做，它的目标用户是谁、核心问题是什么、第一阶段应该做什么、哪些能力必须具备、哪些能力现在不做，以及如何判断产品第一阶段算成功。

### 0.2 本文不解决什么问题

1. 不定义技术架构、数据库表结构、前端目录和状态管理实现。
2. 不定义具体 UI 稿、视觉规范和组件库选型。
3. 不直接拆解开发任务。
4. 不继承旧项目中的代码、状态、接口和实现假设。
5. 不在本阶段定义商业化、账号体系、社群系统等复杂扩展能力。

## 1. 产品背景

### 1.1 问题背景

阅读英文原版 PDF 时，用户通常会遇到以下问题：

1. PDF 原文阅读门槛高，长段落理解困难。
2. 传统翻译方式割裂，用户需要在“阅读器”和“翻译工具”之间来回切换。
3. 重点内容虽然可以截图、复制或记笔记，但整理过程低效且容易丢失上下文。
4. 阅读常常是持续性的，用户需要下次继续从上一次的位置接着看。
5. 在网络不稳定或离线情况下，很多工具无法继续使用，阅读连续性被打断。
6. 普通 PDF 阅读器解决的是“打开文件”，而不是“帮助理解和沉淀内容”。

### 1.2 产品机会

针对英文原版 PDF，有机会做出一个介于“阅读器”和“语言学习辅助工具”之间的产品，让用户在一个界面中完成：

- 阅读原文
- 查看译文
- 查询局部内容
- 高亮重点
- 打标签
- 回看重点
- 继续上次进度
- 逐步建立自己的阅读积累

### 1.3 一句话定义

阅流 ReadFlow 是一个帮助用户持续阅读英文原版 PDF、降低理解门槛并沉淀重点内容的阅读辅助工具。

## 2. 产品目标

### 2.1 核心目标

第一阶段产品要实现以下目标：

1. 用户可以导入英文 PDF 并正常阅读。
2. 用户可以按段落查看原文、译文或双语内容。
3. 用户可以对重点内容进行高亮和简单分类。
4. 用户可以在关闭页面后恢复上次阅读位置。
5. 用户可以缓存已阅读内容，在离线状态下继续查看已缓存内容。
6. 产品能形成一条完整闭环：  
   **导入 → 阅读 → 翻译 → 标注 → 恢复 → 继续阅读**

### 2.2 阶段目标

#### 第一阶段目标（MVP）
完成英文 PDF 阅读主链闭环，并建立稳定的段落、句子、高亮、翻译与恢复基础结构。

#### 第二阶段目标
在 MVP 基础上补强词句理解、重点整理和导出能力。

#### 第三阶段目标
向“长期阅读与知识沉淀工具”扩展，例如单词本、复习、脑图整理、账号体系与多端同步等。

### 2.3 不追求的目标

在第一阶段，本产品不追求：

1. 成为通用文件管理器。
2. 成为复杂笔记软件。
3. 成为社交型读书平台。
4. 成为全能翻译平台。
5. 成为完整的教育课程系统。
6. 覆盖所有 PDF 类型，包括扫描件、图片型 PDF 和极复杂排版文档。

## 3. 目标用户

### 3.1 核心用户

1. 想阅读英文原版书或英文 PDF 材料的人。
2. 英语水平中等，但不希望每句都依赖外部翻译工具的人。
3. 有持续阅读习惯，希望能标出重点、沉淀内容的人。
4. 需要移动场景或离线场景继续阅读的人。

### 3.2 用户特征

1. 不一定英语很好，但愿意接触原版内容。
2. 不满足于“机翻一下就结束”，更希望保持原文阅读体验。
3. 对“阅读连续性”和“重点沉淀”有较强需求。
4. 不喜欢频繁切换工具，希望一个界面完成主要动作。

### 3.3 非目标用户

1. 只想快速扫描文件内容、不需要长期阅读的人。
2. 主要处理扫描件 OCR、复杂排版论文批注的专业编辑用户。
3. 需要多人协作或企业文档工作流的团队用户。
4. 主要阅读中文 PDF 的轻度用户。

---

## 4. 核心使用场景

### 场景 A：首次导入阅读
用户上传一本英文 PDF，系统解析内容，用户可以开始按段落阅读。

### 场景 B：边读边翻译
用户阅读某一段时，希望看到中文译文，或切换成双语对照模式继续读。

### 场景 C：标记重点
用户读到重要段落或某一句时，对其高亮、加颜色，方便以后回看。

### 场景 D：按标签整理
用户希望把高亮内容按主题分类，例如“写作表达”“价值观”“可复用句子”。

### 场景 E：下次继续阅读
用户下次打开文档时，希望自动回到上次阅读位置，并保留阅读模式。

### 场景 F：离线继续阅读
用户提前缓存部分或整本文档后，即使离线，也能继续阅读已缓存内容。

## 5. 产品范围定义

### 5.1 In Scope（第一阶段纳入）

1. PDF 导入
2. 文档解析与阅读展示
3. 段落级内容组织
4. 段落内部句子级辅助单元生成
5. 原文 / 译文 / 双语三种阅读模式
6. 段落级翻译
7. 基础高亮功能（段落级、句子级）
8. 基础标签功能
9. 阅读进度保存与恢复
10. 离线缓存已读内容
11. 错误提示与基础状态反馈

### 5.2 Out of Scope（第一阶段不纳入）

1. 账号登录与云同步
2. 多端实时同步
3. 社区、分享、公开动态
4. 音频朗读 / TTS
5. 复杂导出模板
6. AI 自动脑图
7. 完整单词本系统
8. OCR 扫描件增强
9. 商业化支付体系
10. 多人协作批注
11. 任意字符范围自由拖拽高亮
12. 跨句、跨段自由文本高亮

## 6. 功能需求

---

## 6.1 文档导入与管理

### 6.1.1 PDF 导入

#### 需求
1. 用户可上传本地 PDF 文件。
2. 系统应校验文件格式是否为 PDF。
3. 导入后应生成唯一文档 ID。
4. 系统应记录基础文档信息：
   - 文档 ID
   - 文件名
   - 导入时间
   - 页数（如可获取）
   - 当前状态
5. 导入成功后自动进入解析流程。
6. 导入失败时应给出明确提示。
7. 第一阶段默认支持**文本可提取的英文 PDF**。
8. 对于扫描件、图片型 PDF 或明显不可提取文本的文档，系统应明确提示当前阶段暂不支持或效果有限。

#### 验收标准
1. 用户可以成功导入 PDF。
2. 导入后 UI 有明确状态变化。
3. 错误导入场景可识别并提示。
4. 对不支持或效果有限的 PDF 类型，有清晰反馈，而不是静默失败。

---

### 6.1.2 文档列表与切换

#### 需求
1. 用户可查看已导入文档列表。
2. 用户可切换当前阅读文档。
3. 切换文档时应加载对应阅读状态。
4. 切换失败时不应造成当前阅读界面异常。

---

### 6.1.3 文档删除

#### 需求
1. 用户可删除某个文档。
2. 删除时应同步移除该文档相关本地缓存与阅读状态。
3. 删除操作应要求确认，避免误删。

---

## 6.2 文档解析与阅读展示

### 6.2.1 文档解析

#### 需求
1. 系统应将 PDF 解析为可阅读的文本内容。
2. 阅读展示的主粒度为“段落”。
3. 段落应具备稳定顺序与定位信息。
4. 若解析过程较长，应反馈进度或状态。
5. 若部分内容先完成解析，应允许优先展示已解析部分。
6. 在生成段落和句子单元前，系统应进行基础文本标准化处理，包括但不限于：
   - 换行清理
   - 空白归一化
   - 连字符断词合并（如可识别）
   - 页眉页脚噪声剔除（如可识别）
7. 第一阶段中的“段落”指系统生成的稳定阅读单元，不要求严格等同于 PDF 原始自然段，但必须具备稳定顺序、唯一锚点和可恢复定位能力。
8. 系统应在段落内部生成稳定的句子级辅助阅读单元。
9. 句子切分不以单一标点为唯一依据，而应结合标点、缩写、大小写及上下文规则进行近似识别。
10. 句子切分结果需在同一文档中保持稳定，以支持句子高亮、局部定位、查词上下文和后续恢复。
11. 句子是段落内部派生出的辅助阅读单元，而不是第一阶段的主存储结构。

---

### 6.2.2 阅读界面

#### 需求
1. 用户可按段落阅读内容。
2. 界面应支持清晰区分原文和译文。
3. 长文档不应因一次性渲染全部内容而严重卡顿。
4. 应支持当前阅读区域与上下文内容的连续显示。
5. 在存在句子级高亮时，界面应能正确显示其与段落内容的从属关系，不应破坏段落阅读连续性。

---

### 6.2.3 可见范围管理

#### 需求
1. 系统应维护当前正在阅读的可见内容范围。
2. 当前可见范围用于驱动翻译优先级、缓存优先级和会话保存。
3. 切换位置后应更新当前锚点。
4. 当前可见范围应优先基于稳定段落单元计算，而不是单纯依赖临时滚动位置。

---

## 6.3 阅读模式与翻译

### 6.3.1 阅读模式

系统应支持以下三种模式：

1. 原文模式
2. 译文模式
3. 双语模式

#### 需求
1. 用户可随时切换模式。
2. 切换后阅读位置不变。
3. 当前模式应被保存，下次恢复时继续使用。

---

### 6.3.2 段落级翻译

#### 需求
1. 系统支持对段落进行翻译。
2. 当前可见段落优先触发翻译。
3. 翻译完成后应与原段落稳定绑定。
4. 翻译失败时应提示可重试。
5. 第一阶段默认对当前可见范围及其前后少量缓冲段落执行自动翻译。
6. 对未命中缓存且翻译失败的内容，应展示失败状态与重试入口，但不阻塞原文阅读。
7. 第一阶段的主翻译结果以段落为单位生成和缓存。
8. 翻译结果必须绑定稳定段落单元，不得仅依赖显示顺序或临时渲染文本。

---

### 6.3.3 句子级翻译（第一阶段可选增强）

#### 需求
1. 用户可对某个句子发起局部翻译。
2. 局部翻译不应破坏段落主结构。
3. 句子翻译结果可用于快速理解，而不是替代段落主翻译。
4. 若后续启用句子级翻译，句子结果应绑定稳定句子单元。
5. 句子级翻译不得覆盖或替换段落级主翻译结果。

---

### 6.3.4 目标语言

#### 需求
1. 第一阶段默认支持中文。
2. 系统应保留未来切换目标语言的扩展空间。
3. 若后续增加更多语言，不同语言翻译结果不得互相覆盖。
4. 翻译缓存与结果绑定需包含目标语言维度，避免不同语言结果冲突。

---

## 6.4 高亮与重点标注

### 6.4.1 创建高亮

#### 需求
1. 用户可对段落或句子内容创建高亮。
2. 高亮至少支持颜色区分。
3. 高亮内容应绑定位置与文本快照。
4. 高亮创建后应立即可见。
5. 第一阶段支持：
   - 段落级高亮
   - 句子级高亮
6. 第一阶段中的句子级高亮仅面向系统识别出的单句单元。
7. 第一阶段暂不支持：
   - 跨句高亮
   - 跨段高亮
   - 任意字符范围自由拖拽高亮

---

### 6.4.2 高亮管理

#### 需求
1. 用户可查看当前文档中的全部高亮。
2. 用户可修改高亮颜色。
3. 用户可删除高亮。
4. 高亮删除应立即同步更新展示状态。
5. 高亮应优先绑定到稳定阅读单元（段落 ID / 句子 ID），而不是仅依赖临时显示文本。

---

### 6.4.3 高亮恢复

#### 需求
1. 文档重新打开后，应恢复已有高亮。
2. 若文本结构发生轻微变化，应尽量通过快照或锚点恢复。
3. 高亮应同时绑定结构锚点和文本快照，以提升重新打开文档时的恢复稳定性。
4. 句子级高亮恢复时，应优先定位原句子单元；如句子单元失效，再尝试回退到段落级上下文定位。

---

## 6.5 标签管理

### 6.5.1 标签创建

#### 需求
1. 用户可创建标签。
2. 标签用于分类整理高亮内容。

---

### 6.5.2 标签与高亮关联

#### 需求
1. 用户可将高亮绑定到一个或多个标签。
2. 用户可按标签过滤高亮。
3. 标签点击后可跳转到对应内容位置。
4. 第一阶段中，标签仅绑定高亮，不直接绑定文档或未高亮的段落/句子。
5. 标签跳转应优先回到高亮所属的稳定阅读单元。

---

### 6.5.3 标签管理

#### 需求
1. 用户可重命名标签。
2. 用户可删除标签。
3. 删除标签不应删除高亮本身，只移除关联关系。

---

## 6.6 阅读进度与会话恢复

### 6.6.1 进度保存

#### 需求
系统应保存以下阅读状态：

1. 当前文档
2. 当前阅读位置
3. 当前阅读模式
4. 当前语言
5. 当前标签视图状态（如有）

---

### 6.6.2 会话恢复

#### 需求
1. 用户重新打开文档时，应尽量恢复到上次阅读位置。
2. 恢复时应优先恢复当前文档和阅读锚点。
3. 恢复失败时应允许回退到文档开头或最近可用位置。
4. 会话恢复的优先顺序应尽量为：
   1. 当前文档 ID
   2. 最近阅读锚点
   3. 阅读模式
   4. 当前语言
   5. 标签/过滤状态
   6. 滚动位置微调
5. 会话恢复应优先基于稳定阅读单元，而不是纯像素级滚动位置。

---

## 6.7 离线缓存

### 6.7.1 缓存内容

第一阶段离线缓存应至少覆盖：

1. 文档基础信息
2. 已解析段落原文
3. 已获取的段落译文
4. 高亮数据
5. 标签数据
6. 阅读进度

---

### 6.7.2 缓存任务

#### 需求
1. 系统支持缓存当前文档已读内容。
2. 系统支持缓存整本文档（如性能允许）。
3. 应显示缓存状态与基础进度。
4. 缓存失败时应允许重试。
5. 第一阶段默认优先支持“已解析 + 已查看 + 已翻译”的内容缓存。
6. 整本文档缓存作为增强能力，仅在性能和存储策略明确后纳入实现。
7. 缓存对象需优先绑定稳定阅读单元，而不是纯显示顺序。

---

### 6.7.3 离线阅读

#### 需求
1. 离线状态下，用户至少可继续查看已缓存原文与译文。
2. 离线状态下，不应强依赖后端请求。
3. 对于未缓存内容，应明确提示不可用，而不是空白。
4. 已缓存内容重新打开后，应可按原有段落/句子结构恢复展示。
5. 离线阅读能力的目标是优先保证已缓存内容可读，而不是在第一阶段追求全自动全量离线。

---

## 6.8 错误反馈与状态提示

### 6.8.1 错误类型

系统至少应显式处理以下错误：

1. 导入失败
2. 解析失败
3. 翻译失败
4. 缓存失败
5. 恢复失败
6. 当前文档类型不支持或效果有限
7. 当前内容未缓存，离线不可用

---

### 6.8.2 状态反馈

系统应向用户反馈以下状态：

1. 正在导入
2. 正在解析
3. 正在翻译
4. 正在缓存
5. 离线可读
6. 错误 / 可重试
7. 当前文档暂不支持或解析效果有限
8. 当前内容未缓存

## 7. 非功能需求

### 7.1 响应速度

1. 文档导入后应尽快给出反馈。
2. 阅读模式切换应尽量即时。
3. 已缓存内容读取应快于重新请求。
4. 已缓存文档重新打开时，不应出现明显等待后才看到正文。

### 7.2 稳定性

1. 翻译失败不应导致整本文档不可阅读。
2. 缓存失败不应导致文档本身不可打开。
3. 高亮和标签异常不应导致主阅读流崩溃。
4. 单次段落翻译失败不应影响其他段落的阅读与翻译。
5. 句子切分结果在同一文档中应保持稳定，不应因重复进入页面而频繁变化。

### 7.3 性能

1. 长文档阅读不应出现明显卡死。
2. 不应一次性把所有派生内容重复计算。
3. 本地缓存写入和恢复应有合理节奏，避免阻塞主阅读体验。
4. 长文档不应一次性渲染全部内容。
5. 高亮、翻译和缓存应优先围绕当前可见范围及其邻近内容调度。

### 7.4 可扩展性

后续应允许扩展以下方向：

1. 单词本
2. 导出重点内容
3. AI 总结 / 脑图
4. 云同步
5. 听书 / TTS
6. 多端同步阅读
7. 账号体系
8. 更细粒度局部文本高亮

## 9. 验收标准

### 9.1 功能验收

1. 用户可以成功导入 PDF。
2. 用户可以阅读解析后的段落内容。
3. 用户可以切换原文 / 译文 / 双语模式。
4. 用户可以对重点内容创建和删除高亮。
5. 用户可以创建标签并与高亮关联。
6. 用户关闭页面后可恢复到上次阅读位置。
7. 用户在离线情况下可继续阅读已缓存内容。
8. 用户在翻译、缓存、恢复失败时能得到明确反馈。
9. 段落翻译结果与原文不发生明显错位。
10. 句子级高亮可稳定恢复，不因重复进入文档而大面积错乱。

### 9.2 产品验收

1. 产品主链完整：导入 → 阅读 → 翻译 → 标注 → 恢复。
2. 用户不会因为状态丢失而中断阅读。
3. 用户可以明显感受到它不是普通 PDF 阅读器，而是“帮助理解和沉淀内容”的工具。
4. 第一阶段已建立后续支持句子翻译、查词、单词本和导出的稳定基础结构。

## 10. 风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| PDF 解析结果不稳定 | 阅读结构混乱 | 第一阶段先围绕稳定阅读单元建立结构，不强求完全等同原始自然段 |
| 翻译与原文错位 | 用户信任下降 | 建立稳定锚点、段落绑定和按单元缓存机制 |
| 句子切分不稳定 | 句子高亮、查词、恢复异常 | 句子只作为段落内部辅助单元，采用规则化切分并保持同文档稳定 |
| PDF 原始文本噪声较多 | 翻译不准、切分异常 | 在解析后增加基础文本标准化流程 |
| 离线链路复杂 | 缓存失效或恢复失败 | 第一阶段优先保证已缓存内容可读，不追求全自动复杂逻辑 |
| 状态过多耦合 | 后续难以维护 | 后续通过 Design 明确主状态与子模块边界 |
| 功能做太多 | MVP 失控 | 第一阶段坚持“可阅读闭环优先” |
| 误将扫描件纳入支持范围 | 用户预期失真 | 明确第一阶段仅保证文本可提取 PDF |

## 11. Assumptions & Defaults

1. 本项目为全新项目，不继承旧代码和旧架构。
2. 第一阶段默认聚焦英文原版 PDF。
3. 第一阶段默认目标语言为中文。
4. 第一阶段以 Web/PWA 阅读体验为主。
5. 第一阶段优先保证阅读主链，而不是功能数量。
6. 第一阶段默认支持**文本可提取的英文 PDF**，不保证扫描件、纯图片型 PDF 和极复杂排版 PDF 的完整解析效果。
7. 第一阶段中的“段落”是系统生成的稳定阅读单元，而不是对 PDF 原始文本块的机械映射。
8. 第一阶段中的“句子”是段落内部的辅助阅读单元，用于句子高亮、局部定位和后续理解增强，不作为主存储结构。
9. 第一阶段优先追求结构稳定性和可恢复性，而不是语言学上的绝对切分准确率。
10. 所有翻译、高亮、标签和阅读状态，均应优先绑定到稳定阅读单元，而不是临时渲染文本或纯显示顺序。

## 12. 本文档结论

1. 阅流 ReadFlow 的第一阶段定位，已经明确为“英文 PDF 阅读辅助工具”，而不是通用 PDF 工具。
2. 当前最重要的产品任务，是先建立稳定的阅读、翻译、标注、恢复、离线闭环。
3. 第一阶段已明确段落与句子的层级关系：段落是主阅读单元，句子是辅助阅读单元。
4. 第一阶段已明确高亮边界：支持段落级和句子级高亮，不支持自由拖拽式任意文本高亮。
5. 当前产品需求已经具备进入 Orient 的条件。
6. 本项目后续应基于本文档进入 Orient / Design / Plan，而不是直接从页面或代码实现出发。
