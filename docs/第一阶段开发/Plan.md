## 1. Plan 使用标准

### 1.1 执行地位

一旦进入 `M` 阶段，执行必须以本计划为准，不允许跳过本计划直接做模糊开发。

### 1.2 勾选规则

一条任务只有同时满足以下条件才允许勾选：

1. 正式产物已存在。
2. 本任务要求的验证已完成。
3. 风险与结果已回写。
4. 未留下会破坏后续模块通过条件的明显缺口。

### 1.3 任务粒度规则

1. 一个 Task 只承载一组强关联动作。
2. 一个 Task 执行后必须可判断 `Done / Not Done / Blocked`。
3. 一个 Task 不允许再隐含完整设计问题。
4. 禁止出现“整理一下”“顺手优化”“兼容性处理一下”等模糊表达。

### 1.4 编号规则

1. `Phase`: 阶段目标
2. `Module`: 阶段内的强关联模块
3. `Task`: 可直接执行和验证的动作

示例：

1. `P2`
2. `P2-M1`
3. `P2-M1-T01`

### 1.5 单任务完成定义

每个 Task 至少必须具备：

1. 任务动作
2. 来源 / 引用
3. 前置依赖
4. 预期修改对象
5. 完成标志
6. 验证方式
7. 回写位置

## 4. 全局技术冻结

1. 项目按单仓 Web/PWA 交付。
2. 技术基线为 `Vue 3 + TypeScript + Vite + Pinia + Vue Router`。
3. 段落是主粒度，句子是增强粒度。
4. PDF 首版只承诺文本型 PDF。
5. 本地持久化采用 IndexedDB，应用壳离线采用 Service Worker。
6. 翻译通过 Provider 适配层接入，必须支持 Mock 与降级。
7. 恢复和离线是 MVP 主链，不得后置。
8. `Plan` 不再假设仓库存在既有 `src/store/*` 实现。
