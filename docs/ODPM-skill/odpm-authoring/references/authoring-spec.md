# ODPM 编写规范

> 适用范围：所有采用 `ODPM` 方法组织需求、设计、计划与自动化执行的项目
> 核心原则：`Orient` 负责把目标与边界说清，`Design` 负责把路径与结构说清，`Plan` 负责把执行拆清并卡死，`M` 阶段只负责按批准后的 `Plan` 推进
> 参考标准：本文件定义的是“通用写法与强制流程”；具体项目的真实路径、真实文档和真实口径，必须再结合对应项目补充文件理解
>
> VScode的BUG提示!!!在写入大文件时,必须分批切片写入,否则会写入错误!!!!

## 1. 文档定位

### 1.1 本文件解决什么问题

很多 Agent 会把 `ODPM` 理解成“拿到任务后去执行”的方法，但真正决定结果好坏的，恰恰是执行前的上游文档质量。

本文件只约束一件事：

`Orient / Design / Plan` 该如何被写出来，才能让后续执行阶段不至于偷懒、跑偏、最小化交差。

### 1.2 本文件不解决什么问题

1. 不替代任何项目自己的 `Orient / Design / Plan` 正文。
2. 不替代执行阶段的 `M` 规则；执行时必须再读 `ODPM运行规则.md`。
3. 不替代技术专项规范；写 Python、Flutter、审查计划时仍需回到对应专项规范。

## 2. ODPM 全流程定义

| 阶段       | 核心问题                                 | 产物定位                     | 是否必须用户 Check |
| ---------- | ---------------------------------------- | ---------------------------- | ------------------ |
| `Orient` | 我们到底要做什么，为什么做，边界是什么   | 上游目标与冻结边界文档       | 是                 |
| `Design` | 这个目标准备怎么落地，结构和路径是什么   | 实现蓝图、合同蓝图、验收蓝图 | 是                 |
| `Plan`   | 具体要按什么顺序、什么粒度、什么标准执行 | 自动化执行任务树             | 是                 |
| `M`      | 当前先执行哪一项，怎么验证和回写         | 执行记录与状态推进           | 不在本文件定义     |

## 3. 总体强制流程

任何一轮新的 `ODPM` 编写，都必须按以下顺序推进，不允许跳步：

1. 先明确本轮任务是否真的需要新建或重写 `ODPM`。
2. 若需要，先进入 `Orient`，并与用户协同确认目标、约束、输入资料和完成态。
3. `Orient` 通过用户 Check 后，才能进入 `Design`。
4. `Design` 必须基于 `Orient` 展开，不允许重新发明另一套目标。
5. `Design` 通过用户 Check 后，才能进入 `Plan`。
6. `Plan` 必须把 `Design` 拆到足够细，且每条任务都能直接驱动执行。
7. `Plan` 通过用户 Check 后，才能进入 `M`。
8. 若任一阶段发现上游输入不成立，必须显式回流，而不是带着错误继续往下写。

## 4. 通用写作纪律

### 4.1 先完整阅读，再动手写

编写 `ODPM` 时，禁止只用关键词搜索局部片段就开始写作。执行者必须先完整阅读：

1. 当前项目的项目计划书、项目说明或任务来源文档。
2. 当前项目已有的 `Orient / Design / Plan`。
3. 当前任务涉及的开发规范、合同文档、过程记录、审查结论。
4. 当前代码仓库中与任务主链相关的真实实现。

如果当前项目已经存在经过验证的高质量 `Orient / Design / Plan` 样板，也必须一并阅读，并抽取其章节组织、矩阵写法、冻结口径和任务拆解方式，优先复用成熟写法，而不是重新发明一套更粗糙的结构。

### 4.2 每一层都必须先生成“写作 TodoList”

这是本规范的硬要求。

任何 Agent 在开始写 `Orient / Design / Plan` 前，都必须先生成该阶段的写作 TodoList。
写作 TodoList 不是执行 TodoList，而是“为了把这份文档写对，必须先做哪些阅读、调研、拆解、回归和检查”的清单。

写作 TodoList 至少要覆盖：

1. 输入阅读。
2. 现状盘点。
3. 必要调研。
4. 文档骨架。
5. 风险回归。
6. 用户 Check 准备。

### 4.3 每一层都必须显式写“上游依据”和“采用优先级”

`ODPM` 文档不是纯主观总结。任何 `Orient / Design / Plan` 正文都应在前部写清：

1. 本文的目标文件路径。
2. 版本号。
3. 日期。
4. 状态。
5. 上游依据。
6. 若多份输入之间存在优先级差异，采用什么优先级判断。

### 4.4 每一层都必须有“本文负责什么 / 不负责什么”

这样做的目的是防止文档职责漂移。例如：

1. `Orient` 不负责把代码结构写满。
2. `Design` 不负责把任务切成勾选工单。
3. `Plan` 不负责重新讨论方向。

### 4.5 每一层都必须做回归

1. `Orient` 写完后，要回归用户原始目标与输入。
2. `Design` 写完后，要回归 `Orient`。
3. `Plan` 写完后，要回归 `Orient + Design + 全局输入文档`。
4. 回归结论不允许只写“已检查”，而要写清覆盖关系和风险残留。

## 5. `Orient` 编写规范

### 5.1 `Orient` 的职责

`Orient` 不是实现方案，也不是任务清单。它只负责把“本轮任务为什么做、要做到哪里、不能越过什么边界、后续 Design 必须继承什么冻结点”说清楚。

### 5.2 进入 `Orient` 前必须先做的事

1. 和用户确认本次任务的核心目标。
2. 和用户确认本次任务的现实约束。
3. 和用户确认需要参阅的文档、规范、代码和外部资料。
4. 判断本次任务的完成态是什么。
5. 判断是否存在显式非目标。

如果以上五件事还没有被说清，就不允许直接写 `Design`。

### 5.3 `Orient` 阶段写作 TodoList 模板

开始写 `Orient` 前，至少先列出如下 TodoList：

1. 阅读用户原始要求，摘录“目标 / 约束 / 完成态 / 禁区”。
2. 阅读已有项目计划书、历史决策、过程记录和相关规范。
3. 盘点仓库现实，确认当前代码、目录、文档与目标之间的差距。
4. 提炼本轮 `Orient` 需要冻结的上位结论。
5. 列出仍需用户确认的问题。
6. 按固定章节骨架起草 `Orient`。
7. 做一次“是否越写越像 Design”的回归检查。
8. 输出用户 Check 摘要。

### 5.4 `Orient` 的推荐正文结构

一份合格的 `Orient`，推荐至少包含以下章节：

1. 标题区：
   - 目标文件
   - 版本
   - 日期
   - 用途
   - 上游依据
   - 采用优先级
2. 文档定位：
   - 本文件是什么
   - 本文件不负责什么
   - 高频术语速查
3. 本轮复核结论或输入校核结论。
4. Summary：
   - 一句话定义
   - 当前阶段最重要结论
   - 后续 Design 不得回退的方向
5. 任务背景与当前事实。
6. 核心目标与成功标准。
7. 团队、预算、时间、环境、合规、兼容性等现实约束。
8. `In Scope / Out of Scope`。
9. 对后续 `Design` 的强约束。
10. 冻结点总表。
11. `Design` 阶段必须交付的内容。
12. 结论。

### 5.5 `Orient` 中必须明确写出来的内容

1. 当前任务为什么必须做。
2. 当前任务不是在做什么。
3. 项目真实处境，而不是理想化背景。
4. 成功与失败的判断标准。
5. 后续 `Design` 可以展开什么，不能推翻什么。

### 5.6 `Orient` 的质量闸门

一份 `Orient` 不合格，通常有以下表现：

1. 只有目标，没有约束。
2. 只有背景故事，没有冻结结论。
3. 把技术实现细节提前写进来，导致与 `Design` 职责混淆。
4. 没有 `Out of Scope`，导致后续无限膨胀。
5. 没有成功标准，导致后续无法验收。

### 5.7 `Orient` 完成后的强制回归

写完 `Orient` 后，必须逐项回归：

1. 用户说要做的事，是否都已进入文档。
2. 用户明确不想做的事，是否写进了非目标或边界。
3. 是否已经把本轮任务真正的出发点写清。
4. 是否已经写清后续 `Design` 的输入和禁区。

### 5.8 `Orient` 的用户 Check 要求

提交给用户检查时，至少要请用户确认四件事：

1. 目标理解是否正确。
2. 约束是否漏项。
3. 非目标是否准确。
4. 冻结点是否有明显错误。

没有用户 Check，不允许进入 `Design`。

## 6. `Design` 编写规范

### 6.1 `Design` 的职责

`Design` 的任务，是把 `Orient` 中的目标和冻结点，翻译成一条“能真的走到终点”的实现路径。
它必须让人拿到文档后，不需要你在旁边额外解释，也知道应该怎样搭结构、怎样分层、怎样接线、怎样验收。

### 6.2 进入 `Design` 前必须先做的事

1. 确认 `Orient` 已通过用户 Check。
2. 重新阅读 `Orient`，标出硬冻结点。
3. 盘点现有代码、目录、脚本、测试和合同现状。
4. 判断本轮是否涉及外部库、协议、算法、框架或近期会变化的信息。
5. 若涉及，先做调研，不允许凭模糊记忆写 Design。

### 6.3 `Design` 阶段写作 TodoList 模板

开始写 `Design` 前，至少先列出如下 TodoList：

1. 通读 `Orient`，整理本轮不可回退的冻结点。
2. 扫描现有代码和目录，确认哪些现实必须被 Design 承接。
3. 阅读相关编码规范、合同规范、部署规则、审查结论。
4. 对涉及的库、框架、协议、算法做必要调研，并记录是否适配本项目。
5. 设计阶段划分、模块划分、职责边界和接口边界。
6. 设计状态机、通信链路、数据归属和配置切换口径。
7. 设计每个阶段和模块的完成定义、验收标准和 Gate。
8. 做一次“是否完整覆盖 Orient”的回归。
9. 准备用户 Check 摘要。

### 6.4 `Design` 的推荐正文结构

一份合格的 `Design`，推荐至少包含以下章节：

1. 标题区：
   - 目标文件
   - 版本
   - 日期
   - 状态
   - 上游依据
   - 采用优先级
   - 文档目标
2. 文档标准：
   - 本文必须达到的标准
   - 本文负责什么
   - 本文不负责什么
3. 核心开发结论。
4. 环境矩阵、模式矩阵、依赖矩阵。
5. 工作区与代码落位设计。
6. 分层、SOLID 约束与可改动边界。
7. 启动脚本、调试策略、CI、迁移与回滚纪律。
8. API 总面设计。
9. 数据与字段设计。
10. 状态机与控制流设计。
11. 各业务域实现设计。
12. 分阶段执行路径。
13. Gate 与统一完成定义。
14. 结论。

### 6.5 `Design` 中必须出现的“清晰化产物”

以下内容不能只口头提，要尽量用表格、矩阵或编号冻结：

1. 环境矩阵。
2. Provider 或外部能力的 mock / real 切换矩阵。
3. 目录落位图。
4. 模块职责表。
5. 状态归属表。
6. API 或资源清单。
7. 关键状态机。
8. 分阶段 Gate。
9. 统一完成定义。

### 6.6 `Design` 的外部调研要求

当任务涉及外部库、协议、框架、算法、标准、SDK 或最新行为时，必须执行调研，并在文档中体现“为什么这个方案适合当前项目，而不是只是在别的项目里常见”。

至少要回答：

1. 这个方案依赖什么前提条件。
2. 当前项目是否满足这些前提条件。
3. 若不完全满足，需要做什么裁剪或改造。
4. 有哪些不采用的备选方案，以及为什么不选。

### 6.7 `Design` 的质量闸门

一份 `Design` 不合格，通常有以下表现：

1. 只有宏观思路，没有工程路径。
2. 只说“采用某技术”，但没写为什么适合当前项目。
3. 只说“后续补测试 / 后续接真源”，把关键问题推后。
4. 模块职责、状态归属、接口边界仍然混乱。
5. 没有 Gate，无法判断阶段是否真的完成。

### 6.8 `Design` 完成后的强制回归

写完 `Design` 后，必须逐项回归：

1. 是否完整覆盖了 `Orient` 的目标。
2. 是否遵守了 `Orient` 的边界和非目标。
3. 是否明确写出了实现顺序、结构路径和阶段验收。
4. 是否存在明显走不通、会引发大返工或逻辑冲突的设计。

### 6.9 `Design` 的用户 Check 要求

提交给用户检查时，至少要请用户确认四件事：

1. 这条路线是否符合预期。
2. 分阶段和分模块是否合理。
3. 是否存在用户明确不接受的设计方向。
4. 是否可以进入 `Plan` 拆解。

没有用户 Check，不允许进入 `Plan`。

## 7. `Plan` 编写规范

### 7.1 `Plan` 的职责

`Plan` 是自动化执行前的最后一道硬约束。
从 `M` 阶段开始，Agent 会严格根据 `Plan` 选任务、做任务、打勾；因此 `Plan` 的粒度、字段、完成定义、验证动作和来源挂钩，直接决定最后结果。

### 7.2 进入 `Plan` 前必须先做的事

1. 确认 `Design` 已通过用户 Check。
2. 重新阅读 `Orient + Design + 全局输入文档`。
3. 提炼出正式 Phase、Module、Gate 和冻结约束。
4. 判断哪些内容已经足够稳定，能拆到执行粒度。
5. 若某块设计仍然模糊，不允许硬拆 `Plan`，必须先回流到 `Design`。

### 7.3 `Plan` 阶段写作 TodoList 模板

开始写 `Plan` 前，至少先列出如下 TodoList：

1. 通读 `Orient + Design + 全局输入文档`，整理 Phase / Module / Gate。
2. 提炼所有全局冻结点，并决定哪些要写成“全局技术冻结”。
3. 编写“来源追溯矩阵”，说明计划约束从哪里来。
4. 先按 Phase 拆，再按 Module 拆，最后按 Task 拆。
5. 为每个 Module 写目标、过程索引、通过条件。
6. 为每个 Task 写编号、动作、完成标志、验证方式。
7. 检查是否遗漏脚本、CI、迁移、文档回写、测试和验收动作。
8. 专门做一次粒度审查，确认没有“整理一下”“优化一下”这类句子。
9. 专门做一次覆盖审查，确认 `Orient + Design` 的内容都能在 Plan 中找到落点。
10. 准备用户 Check 摘要。

### 7.4 `Plan` 的推荐正文结构

一份合格的 `Plan`，推荐至少包含以下章节：

1. 标题区：
   - 目标文件
   - 版本
   - 日期
   - 状态
   - 上游依据
   - 采用优先级
   - 文档目标
2. Plan 使用标准：
   - 执行地位
   - 勾选规则
   - 拆解粒度规则
   - 编号规则
   - 单任务完成定义
3. Phase 总览。
4. 全局技术冻结或全局执行冻结。
5. 过程记录 / 输入文档追溯矩阵。
6. 各 Phase 详细任务。
7. 各 Gate 通过条件。
8. 结束条件。
9. 覆盖率复核。

### 7.5 `Plan` 的推荐任务结构

对于大型项目，推荐采用“`Phase -> Module -> Task`”三级结构：

1. `Phase`：
   - 表示一个阶段性目标，如工作区重组、合同落盘、服务端底座、客户端迁移。
2. `Module`：
   - 表示一个强关联的子系统或阶段模块。
   - 必须写：
     - 过程记录索引或来源索引
     - 模块目标
     - 模块通过条件
3. `Task`：
   - 表示一个可直接执行、可直接验收、可直接勾选的动作。
   - 必须写：
     - 任务编号
     - 明确动作
     - 完成标志

### 7.6 `Task` 的强制字段要求

`Plan` 里的每条任务，至少要能回答下面这些问题；如果不适合全部写在一行任务里，就必须在模块描述或任务卡片里补齐：

1. 这条任务到底做什么。
2. 它的上游来源是什么。
3. 它依赖哪些前置条件。
4. 它改哪些东西。
5. 做到什么程度才算完成。
6. 用什么方式验证。

推荐字段如下：

1. 任务编号。
2. 任务动作。
3. 来源 / 引用。
4. 输入资料。
5. 前置依赖。
6. 执行边界。
7. 预期修改对象。
8. 产物。
9. 完成标志。
10. 验证方式。

### 7.7 `Plan` 的粒度规则

`Plan` 不能太粗，也不能碎成机械步骤。判断标准如下：

1. 一条任务只承载一组强关联动作。
2. 该任务执行完后，应当能给出明确的“已完成 / 未完成 / BLOCKED”结论。
3. 该任务不应再隐含一个完整的新设计问题。
4. 该任务不应大到让 Agent 能用最小实现随便交差。

### 7.8 `Plan` 中允许和不允许的写法

允许的写法：

1. “新建 `server/infra/scripts/dev/run_worker_wsl.sh`，完成标志：worker 可通过统一脚本启动”
2. “改写 `.vscode/launch.json` 指向新客户端入口，完成标志：旧根目录 `lib/main.dart` 不再被正式引用”
3. “补 `BillingOrder` 错误码词典，完成标志：相关字段和错误码已进入 `docs/contracts`”

不允许的写法：

1. “整理目录”
2. “搭服务端架构”
3. “优化客户端”
4. “做兼容性”
5. “把联调问题处理一下”

### 7.9 `Plan` 粒度审查清单

`Plan` 写完后，必须专门做一轮粒度审查。至少检查：

1. 是否仍存在模糊词。
2. 是否仍有大任务没有拆开。
3. 是否遗漏启动脚本、验证、文档回写、测试或 CI。
4. 是否有任务写完后仍然无法判断算不算完成。
5. 是否有任务只给动作，没有完成标志。

### 7.10 `Plan` 覆盖性回归清单

`Plan` 写完后，必须反向回归：

1. `Orient` 的目标、边界、非目标是否都被承接。
2. `Design` 的阶段、模块、Gate、结构约束是否都被拆解。
3. 全局输入文档的关键约束是否已挂接到某个 Module 或 Task。
4. 是否还存在“大家脑中默认知道，但 Plan 没写出来”的关键约定。

### 7.11 `Plan` 的用户 Check 要求

提交给用户检查时，至少要请用户确认：

1. 大阶段是否完整。
2. 模块拆解是否合理。
3. 任务粒度是否足够细。
4. 关键来源和约束是否挂钩清楚。
5. 完成标志和验证方式是否足够严格。

没有用户 Check，不允许进入 `M`。

## 8. 编写阶段必须产出的 TodoList 规则

这是本规范额外强调的硬约束。

### 8.1 `Orient` 写作 TodoList 的最低要求

必须覆盖：

1. 用户目标澄清。
2. 输入资料收集。
3. 仓库现实盘点。
4. 冻结点提炼。
5. 非目标提炼。
6. 用户 Check 准备。

### 8.2 `Design` 写作 TodoList 的最低要求

必须覆盖：

1. `Orient` 冻结点摘录。
2. 代码与目录现状扫描。
3. 技术规范与合同规则阅读。
4. 外部调研。
5. 阶段与模块设计。
6. 风险与 Gate 设计。
7. 用户 Check 准备。

### 8.3 `Plan` 写作 TodoList 的最低要求

必须覆盖：

1. `Orient + Design + 全局输入文档` 消化。
2. Phase / Module / Gate 拆解。
3. 来源追溯矩阵整理。
4. 任务编号和完成标志设计。
5. 粒度审查。
6. 覆盖率回归。
7. 用户 Check 准备。

### 8.4 TodoList 的使用要求

1. 没有 TodoList，不允许开始写正文。
2. TodoList 不是形式主义；如果中途发现缺项，必须补齐。
3. TodoList 必须服务于“写对这份文档”，而不是服务于“赶紧把文档写完”。

## 9. 从 `Plan` 交接到 `M` 的条件

只有同时满足以下条件，才能认为这轮 `ODPM` 已经具备自动化执行条件：

1. `Orient` 已完成并通过用户 Check。
2. `Design` 已完成并通过用户 Check。
3. `Plan` 已完成并通过用户 Check。
4. 相关规范、合同、过程记录、项目补充文件已挂接清楚。
5. 执行者拿到 `Plan` 后，不需要重新设计主链，只需要按计划执行。

## 10. 禁止事项

1. 禁止没有 `Orient` 就直接写 `Design`。
2. 禁止没有 `Design` 就直接拆 `Plan`。
3. 禁止跳过用户 Check。
4. 禁止在文档里大量使用模糊句式。
5. 禁止只写原则，不写结构、模板、完成定义和回归。
6. 禁止只盯局部任务，不看项目全局输入。
7. 禁止把“先写起来再说”当成 `ODPM`。
