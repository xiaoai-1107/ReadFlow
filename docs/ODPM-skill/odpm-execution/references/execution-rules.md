# ODPM 运行规则

> 适用范围：已经具备可执行 `Orient / Design / Plan` 的项目
> 核心定位：本文件只负责 `M` 阶段，也就是按批准后的 `Plan` 严格执行、验证、回写、打勾的过程
> 关键原则：执行者不是来重新设计主链的，而是来按 `Plan` 把任务做完；若发现 `Plan` 不够支撑执行，应回流上游，而不是现场自由发挥

## 1. 文档定位

### 1.1 本文件负责什么

本文件只定义执行阶段的标准动作：

1. 执行前先读什么。
2. 如何选择任务。
3. 如何生成执行 TodoList。
4. 如何实施、验证、回写、打勾。
5. 什么情况下必须回流到 `Orient / Design / Plan`。

### 1.2 本文件不负责什么

1. 不负责教你如何写 `Orient / Design / Plan`；那是 `ODPM编写规范.md` 的职责。
2. 不负责替具体项目决定真实文档路径；那是项目补充文件的职责。
3. 不负责替技术专项规范做裁决；写 Python、Flutter、审查时仍要读对应专项规范。

## 2. 进入 `M` 阶段的前置条件

执行者只有在以下条件全部成立时，才能进入 `M`：

1. 已有正式的 `Orient`。
2. 已有正式的 `Design`。
3. 已有正式的 `Plan`。
4. 上述三者都已经过用户 Check。
5. 当前项目的补充规则、合同和专项规范可被定位。

若以上任一条件不成立，执行者不得硬进 `M`，而必须回到 `ODPM编写规范.md` 补齐上游。

## 3. 执行前的强制阅读顺序

执行者在真正改代码或改文档前，必须按顺序完整阅读，而不是只搜任务编号：

1. `docs/开发规范/README.md`
2. `docs/开发规范/ODPM编写规范.md`
3. 本文件
4. 当前项目的 `ODPM` 项目补充
5. 当前任务涉及的 `Orient`
6. 当前任务涉及的 `Design`
7. 当前任务所在的 `Plan`
8. 当前任务引用到的合同、字段字典、错误码、过程记录、专项规范

### 3.1 禁止只看局部任务

执行 `P3-M2-T04` 这类任务时，禁止只读取这一行任务本身。  
至少还必须读：

1. 所在 `Phase` 的目标。
2. 所在 `Module` 的目标与通过条件。
3. 当前 `Plan` 的全局冻结与追溯矩阵。
4. 对应的 `Design` 章节。
5. 对应的 `Orient` 冻结点。

原因很简单：

只看一行任务，Agent 很容易把任务做成“局部最小修补”；只有把模块目标、Gate 和上游边界一起看，执行结果才不会跑偏。

## 4. `M` 阶段的标准执行流程

任何执行者在 `M` 阶段都必须按以下流程推进：

1. 选择一个明确的 `Task`。
2. 阅读该任务所属 `Module` 与 `Phase` 的上下文。
3. 回查 `Orient / Design / 合同 / 规范 / 过程记录`。
4. 为本次任务生成执行 TodoList。
5. 按 TodoList 实施任务。
6. 按 `Plan` 要求做验证。
7. 回写结果、风险、阻断、影响面和文档变更。
8. 只有在“产物存在 + 验证通过 + 回写完成”同时成立时，才能打勾。
9. 再进入下一条任务。

## 5. 任务执行包规则

在真正实施当前任务前，执行者必须先整理出本次任务的“执行包”。  
执行包的作用，是把本次任务真正需要的上下文、边界、验证和回写要求压成一个结构化载体，防止执行者只抓住任务标题就动手。

### 5.1 执行包至少应包含什么

1. 当前任务编号。
2. 所在 `Phase / Module`。
3. 模块目标与通过条件摘要。
4. 对应的 `Orient` 冻结点。
5. 对应的 `Design` 路线或章节。
6. 当前任务引用到的合同、规范、过程记录。
7. 当前任务的前置依赖。
8. 受影响的目录、文件、脚本、测试或文档。
9. 当前任务的完成标志。
10. 当前任务的验证方式。
11. 当前任务的回写位置。

### 5.2 执行包与 TodoList 的关系

1. 执行包回答“这次任务到底站在什么上下文里做”。
2. 执行 TodoList 回答“这次任务准备按什么步骤做”。
3. 两者缺一不可；没有执行包，TodoList 往往只剩实施动作，没有边界和来源。

## 6. 任务选择规则

### 6.1 一次只选一个明确子任务

执行时，默认一次只处理一个 `Task`。  
即使用户要求批量推进，也必须按“单任务循环”逐项完成，而不是把多个无关任务混成一次模糊改动。

### 6.2 选任务时必须检查依赖

在真正执行前，必须判断：

1. 当前任务依赖的前置任务是否已经完成。
2. 当前任务是否会改动模块通过条件中的关键对象。
3. 当前任务是否受某条全局冻结约束。
4. 当前任务是否触发专项规范。

若依赖未满足，不允许强做，应标记 `BLOCKED` 或回流。

## 7. 执行 TodoList 规则

这是 `M` 阶段的核心硬约束。

### 7.1 没有执行 TodoList，不允许开始实施

每次执行单个任务前，必须先生成本次任务的执行 TodoList。  
执行 TodoList 用来约束执行者“这次具体要读什么、做什么、怎么验、怎么回写”，避免 Agent 直接跳到修改代码阶段。

### 7.2 执行 TodoList 的最低结构

执行 TodoList 至少必须覆盖以下六类动作：

1. 读取依据：
   - 当前任务所在 `Phase / Module / Task`
   - 对应 `Orient / Design`
   - 对应合同或专项规范
2. 影响面扫描：
   - 需要看的目录
   - 需要看的文件
   - 上下游依赖
3. 具体实施：
   - 计划修改的代码、文档、脚本或配置
4. 验证动作：
   - 测试
   - 检查命令
   - 人工验证
5. 回写动作：
   - `Plan` 状态
   - 文档
   - 运行说明
   - 风险说明
6. 收口判断：
   - 什么情况下可以打勾
   - 什么情况下只能写 `BLOCKED`

### 7.3 执行 TodoList 模板

推荐最少写成下面这种形式：

1. 读取任务与模块上下文。
2. 回查对应 `Orient / Design / 合同 / 专项规范`。
3. 扫描受影响的代码和文档。
4. 实施本任务要求的改动。
5. 跑本任务要求的验证。
6. 回写文档、风险与任务状态。

### 7.4 TodoList 不允许的写法

以下写法都不合格：

1. “顺手做点相关优化”
2. “看看还有什么一起改了”
3. “先把主逻辑写出来，测试后面补”
4. “有空再回写文档”

这些句子都会让执行重新滑回“最小化交差”。

## 8. 具体实施规则

### 8.1 必须按任务边界实施

执行者只能做当前任务要求的内容。若在实施过程中发现额外问题：

1. 若是完成当前任务所必需的依赖修复，可在本任务内一并处理，但必须在回写中说明。
2. 若属于新的主链问题，不允许“顺手捎带做掉”，必须回到 `Plan` 体系处理。

### 8.2 必须抵抗“最小解释”

Agent 在执行中天然会倾向：

1. 选最省事解释。
2. 只做最少改动。
3. 尽快让任务看起来“差不多完成”。

因此，执行时必须主动检查：

1. 当前理解是否只是这条任务的最小解释。
2. 当前实现是否真的满足了模块目标和完成标志。
3. 是否漏掉了脚本、文档、验证、CI、兼容性、回写这类非代码动作。

### 8.3 禁止跳过上游约束

即使当前代码改动看起来很小，也不能跳过：

1. 全局冻结。
2. 模块通过条件。
3. Gate 条件。
4. 专项规范。

小改动同样可能破坏上游设计。

## 9. 验证规则

### 9.1 没有验证证据，不允许打勾

验证证据可以是：

1. 命令输出。
2. 测试通过。
3. 人工操作结果。
4. 结构校对结果。
5. 文档回写结果。

### 9.2 验证必须对齐 `Plan`

验证不能靠执行者自己拍脑袋决定，而必须对齐当前任务在 `Plan` 中写明的完成标志和验证方式。

### 9.3 验证至少要回答三件事

1. 产物是否真的存在。
2. 任务要求是否真的成立。
3. 上下游依赖是否没有被破坏。

### 9.4 验证失败时的处理

1. 若问题可以在当前任务边界内修复，就继续修到满足要求。
2. 若问题暴露出 `Plan` 粒度不足、设计冲突或上游错误，必须回流。
3. 若环境缺失导致无法验证，必须明确写 `BLOCKED`，不能口头算完成。

## 10. 回写规则

### 10.1 每条任务都必须回写

回写内容至少应包括：

1. 做了什么。
2. 改了哪些对象。
3. 用什么验证。
4. 验证结果如何。
5. 是否有风险、已知限制或阻断。
6. 是否同步更新了文档、脚本、合同或计划状态。

### 10.2 回写位置必须可追踪

执行结果要回写到正确位置，例如：

1. `Plan` 勾选状态。
2. 对应设计或运行文档。
3. 审查报告。
4. 合同文档。
5. 运行说明或 README。

### 10.3 回写不是可选项

没有回写，即使代码写完、测试通过，也不能算任务完成。

## 11. 打勾规则

### 11.1 只有同时满足以下条件，才能打勾

1. 当前任务要求的正式产物已经存在。
2. 当前任务要求的验证已经完成。
3. 必要回写已经完成。
4. 当前任务没有留下会破坏模块通过条件的明显缺口。

### 11.2 以下情况严禁打勾

1. 只写了代码，没验证。
2. 只验证了局部，没检查上游影响。
3. 只改了文档，但任务要求的是正式代码或脚本产物。
4. 还存在明确阻断，只是“感觉差不多了”。
5. 还没补回写。

## 12. 回流机制

执行中若发现以下情况，必须回流，而不是继续硬做：

1. 当前任务与 `Orient` 明显冲突。
2. 当前任务无法从 `Design` 中找到清晰路径。
3. 当前 `Plan` 的任务粒度过粗，无法明确边界与完成标准。
4. 当前任务依赖的外部条件与上游文档不一致。
5. 继续执行会引发结构性返工。

### 12.1 回流时必须记录什么

1. 问题发生在哪个任务。
2. 冲突的上游依据是什么。
3. 建议回流到 `Orient`、`Design` 还是 `Plan`。
4. 是否需要重新做用户 Check。

### 12.2 不允许的回流替代动作

1. 不允许自己偷偷改小目标。
2. 不允许自己重写主链再假装是执行。
3. 不允许继续做一个最小版本交差。

## 13. 批量执行规则

### 13.1 允许批量推进，但不允许批量模糊化

用户要求“继续做下一批任务”时，执行者可以连续处理多个任务，但仍必须：

1. 一条任务一条任务地生成执行 TodoList。
2. 一条任务一条任务地实施、验证和回写。
3. 一条任务一条任务地决定是否打勾。

### 13.2 不允许跨模块乱跳

若当前模块尚未收口，默认不应跳到另一个无关模块。  
只有在 `Plan` 本身明确允许并行，或用户明确要求调整顺序时，才可切换。

## 14. 禁止事项

1. 禁止跳过 `Orient / Design / Plan` 直接进入执行。
2. 禁止只靠关键词搜索和局部文件猜测任务要求。
3. 禁止没有执行 TodoList 就直接动手。
4. 禁止把“最小可运行”偷换成“符合 Plan”。
5. 禁止未回写先打勾。
6. 禁止发现 `Plan` 不足时仍然硬做。
7. 禁止让计划、文档和代码长期分离。
