# 移动端与 PC 端句子交互问题分析

## 1. 背景
当前英文阅读网站在 **PC 端与移动端** 的句子交互、卡片展示、结构导航、高亮标注与翻译链路上存在一批明显问题，涉及 UI 割裂、状态失效及排版灾难。本项目旨在遵循“先分析、再实操代码”的原则，针对 6 个核心交互问题进行双端归因，输出可操作的修复建议。

## 2. 范围
本报告重点分析并覆盖以下 6 个核心问题：
1. 句子卡片贴底后底部遭裁切、不跟随滚动
2. 句子高亮区域识别与挂载错误（移动端尤为严重）
3. Structure 面板过长掩盖 My Study
4. 翻译功能不可用与“假翻译”现象
5. 点击外部区域无法关闭句子卡片
6. 高亮标注与现有标签操作体系未打通

## 3. 当前相关模块梳理
- **主视图与页面级结构**：`src/pages/ReaderView.vue` (负责双栏布局与移动端 Action Sheet / Drawer)
- **事件交互**：`src/utils/readerInteractions.ts` 及 `ReaderView.vue` 内部事件绑定（负责点击和高亮切割）
- **数据管层**：`src/stores/reader.ts`、`src/stores/highlight.ts`（支持假翻译拦截和 tags 联动）
- **翻译服务**：`src/services/translation/index.ts`（翻译请求出口及异常捕获层）

## 4. 问题 1 分析：句子卡片贴底后不会跟随句子移动，底部场景卡片展示不完整

### 4.1 现象
点击靠近底部的句子时，弹出的句子卡片出现于底部且极易被裁切；页面滚动时，卡片如“幽灵”般停在屏幕固定位置，脱离当前句子；移动端时常叠加避让遮挡。

### 4.2 影响范围
- **全端**：点击段落特定句子激活的 `sentence-card` (Sentence Card UI层)

### 4.3 当前实现
由 `ReaderView.vue` 内 `sentenceSelectionStyle` 负责卡片样式算子控制：
在生成卡片状态时，代码直接将点击事件自带的纯静态屏幕坐标 (`event.clientX` 和 `event.clientY`) 赋给卡片的 `left\top` （作为 `position: fixed` 的定盘），之后再也没有任何对 `scroll` 的联动侦听。
对移动端，其 CSS 被强制覆盖为 `bottom: calc(...)`，使用 100dvh 计算，卡片高度被撑爆而突破可视范围。

### 4.4 根因候选
1. **[首要] 缺少滚动与元素位置双向绑定机制**：只有一次性的初次点击坐标，未使用 `getBoundingClientRect()` 并随视口、容器 scroll / resize 更新定位。
2. **[次要] 无碰撞与反转检测**：底层缺失类似 `Popper.js / Floating UI` 范式的可视区域边界检测，到底时不会自动向上翻转（flip）。
3. **[移动端专项] 高度计算黑洞**：移动端强制将卡片置底定死（`bottom`），但高度最大值（max-height）受安全区（`safe-area-inset-bottom`）、工具栏和动态虚键的影响未能彻底减除正确参数而生生被裁。

### 4.5 双端差异
- **PC 端**：固定坐标跟随丢失是绝对痛点；溢出裁切为底部命中无翻转策略所致。
- **移动端**：根本用不到 `clientX/Y`，而是依赖纯 CSS `fixed bottom`，所以不是“不跟随”的问题，而是视口高度运算失真（导致了裁切），外加强行脱离句子视线阻断阅读连贯性。

### 4.6 修复建议
- **PC端**：引进浮动定位组件（如 `@floating-ui/vue`），将弹层 `reference` 锚入高亮的 DOM `span`，开启 `autoUpdate`（包含 scroll / resize 刷新）及 `flip` (下沿碰撞时自动上翻)。
- **移动端**：重构底部弹层的 CSS 避让规则，弃用易产生误差的 `dvh/vh` 测算硬堆叠，考虑将卡片轻量化为“纯挂载工具条”并在详情页中展开。

## 5. 问题 2 分析：句子高亮区域错误，移动端更严重

### 5.1 现象
高亮底色背景脱靶，未能按文字实际流线形状严丝合缝地覆盖。多行文本下常变成一个庞大的“直角矩形块”，在越窄的屏幕（移动端）下断行越多，错位越扎眼。

### 5.2 影响范围
- 正文渲染层 `article.paragraph` 及其内嵌的各个 `button.sentence-fragment`。

### 5.3 当前实现
PDF只做纯文本解析。文字在 DOM 中的载体为 `<button class="sentence-fragment">`。选中间态和持久态高加亮则是利用 `.marked` 类增加一个 `background: linear-gradient(...)`。

### 5.4 根因候选
1. **[致命] 滥用 `<button>` 承载普通正文文本而丢失 `display: inline`**：由于 `<button>` 元素浏览器默认表现为 `inline-block` 甚至置换特性，对于它的换行流，不会生成多行 `inline` 切割矩形盒，而是直接将包裹所有文字的两级边界定做一个四方大盒铺设渐变背景，导致高亮直接连片甚至吞噬附近文本。
2. **[次要] 切分容差**：`splitSentences` 的正则导致空格和标点成为单独的块或脱离包裹。

### 5.5 双端差异
问题纯粹共性发生。但由于**移动端可视宽度小**，单句断行、跨行的频次呈数倍提升。而跨行越多，`inline-block` 盒模型所导致的畸形巨型矩形就越明显、错位越惨烈。

### 5.6 修复建议
- 废弃对 `button` 标签的排斥性高亮处理。将 `.sentence-fragment` 改为纯 `<span>`，并严格给它赋予 `display: inline`，使 CSS 背景颜色能忠实匹配真实流式折行；
- 若须保留 button 属性做全局可访问区，则建议将内容使用 CSS Custom Highlight API 实现真实的 Selection 级跨行绘画。

## 6. 问题 3：左侧 Structure 太长，占用篇幅过大，使得 My Study 难直达

### 6.1 现象
展开左侧边栏时，全部的骨架/标题节点一字排开且无底洞蔓延，导致与其在同级纵向排布的 `My Study` 和操作区只能被迫置于最底端，需狂拉滚轮才能看到。

### 6.2 影响范围
- PC 端的 `aside.sidebar` (及其内侧 `Structure` 和 `My Study` 的 `section.panel`)
- 移动端的 Action Sheet 关联拉出的 `Structure Drawer`。

### 6.3 当前实现
侧栏逻辑简单粗暴：在一个父级纵向 Stack (CSS flex 侧向排列或 block 排列) 里，直接用两个 `<section class="panel">`。但对 `structureItems` 的渲染没有限定局部的高度容器 (高度不封顶)，也没有父子节点的层级折叠逻辑。

### 6.4 根因候选
1. **缺少独立滚动域设计**：所有的列表均共享 `sidebar` 的全局纵向滚动。
2. **缺少虚拟列表或折叠/Tab控件**：全量渲染结构长名单，且无手风琴（Accordion）折叠分组收纳。

### 6.5 双端差异
- **PC端**：核心灾区。属于“上下叠放”却无限制。用户无法直接跨越数十个页面的 Structure 而点进 My Study。
- **移动端**：使用了 `readerTools` 的网格菜单（抽屉），在独立 Drawer 里 Structure 和 My Study 是分属不同入口按键。它在移动端的触达上反而是平衡的，但 Structure 内部自己仍是超长。

### 6.6 修复建议
- **架构重构 (PC)**：将 `Structure` 和 `My Study` 由“上下游叠加式”改为“Tabs 标签页平行式”，确保首屏切换效率；或保留上下层并给予 `Structure` 固定 `max-height` (如 `flex: 1`与 `overflow: auto`) 让其局域滚动。

## 7. 问题 4 分析：当前翻译全部是假的

### 7.1 现象
翻译模式下，翻译完全未接通，用户不仅得不到正确译文，还会看见不可用的 fallback 兜底占位文案，实际业务彻底割裂。

### 7.2 影响范围
- 所有选中的原文点击卡片 (`sentence-card-meaning`) 及 `译文模式` 视图。

### 7.3 当前实现
经翻阅开发记录 `17-工具栏语言切换与真实翻译修复记录.md` 得知：为了不“隐瞒”假数据，第一阶段代码对 `stores/reader.ts` 和 `translation/index.ts` 主动执行了整改：**一旦没切中真实 `endpoint`，翻译请求直接返回 `unavailable`。**
前端判定 `!hasLiveTranslation()` 时，由于数据源全盘锁死，强行回退展示：`"No live translation provider is configured..."`。

### 7.4 根因候选
1. 此属于**业务联调缺位问题**。“假翻译”确实被治理了，不再以 mock 欺骗用户，但这等同于功能空载。本地/线上并未实装一个可运行的真网关 Endpoint (即缺乏实际的 LLM 节点与 ENV 参数导入)。
2. 后段请求 `fetch` 接口在没有 .env 支持时，强行让请求 `throw new TranslationServiceError`。

### 7.5 双端差异
纯链路与业务状态数据问题，与端无关。双端均走向同一 `hasLiveTranslation` 断言并失败。

### 7.6 修复建议
- 补齐真实后端联调，补充环境配置映射。
- 即便退一步提供占位体验作“内测”，也不应展示冷冰冰的 developer error 报告；应封装一个提供真正本地离线字元映射的小型 mock dictionary，在 UI 上显眼地申明“当前工作于离线模拟层”。

## 8. 问题 5 分析：点击句子弹出卡片后，点击卡片以外区域应关闭卡片

### 8.1 现象
点击句子呼出了卡片。但当用户期望通过点击周围空白处来注销卡片状态时，由于点击事件完全穿透或者未被监听，卡片“僵死”在屏幕中央无法关闭。

### 8.2 影响范围
- 全局交互与页面背景容器（`main.content` / `rf-page-shell`）。

### 8.3 当前实现
在 `ReaderView.vue` 内部，有着非常完善的逻辑实体——函数 `handleDocumentPointerDown(event)` 和清空选区的核心方法 `resetSelection()`；
但**该函数全局没有任何一处被绑定执行**（`onMounted` 中仅绑定了窗口和视觉事件，极其戏剧性地将 click outside 绑定的代码行遗漏）。这导致点击外层的逻辑成为无根之水。

### 8.4 根因候选
1. **事件侦听死角**：缺少 `window.addEventListener('pointerdown', handleDocumentPointerDown)`，直接由于人为失误导致函数不调配。
2. **组件冒泡抑制**：弹窗上的 `@click.stop` 是有的，但背景层没有主动捕获。

### 8.5 双端差异
- **PC 端**：鼠标点击除了选中其它句子以外的操作均无法解解卡片停滞，必须手动点 "Close" 按钮。
- **移动端**：在打开了独立 Action Sheet 的前提下（如它有单独的 Overlay 遮罩），点击其遮罩可逃脱；但在正文纯句子上高亮唤起的常驻卡片时，也陷入死局。

### 8.6 修复建议
- 将 `handleDocumentPointerDown` 的监听绑定复原至 `onMounted` 及对应释放于 `onBeforeUnmount` 中。
- 清理判断链中无益的 `target.closest` 拦截黑洞，或引入标准的指令如 `v-click-outside` 实施更加稳妥的目标 DOM 切分。

## 9. 问题 6 分析：句子高亮要支持标记现有标签

### 9.1 现象
现有的句子交互流程中，高亮归高亮、贴标签归贴标签。在卡片上必须二次点击 `editTags` 才能“艰难地”接触到老标签（Existing Tags）；交互极其不直观，无法复用老动作来高频流转。

### 9.2 影响范围
- 句子卡片区域内的 `.sentence-card-actions` 与 `.tag-editor-panel`。

### 9.3 当前实现
数据结构虽已充分将 `TagRecord` 能够映射多个 `highlightIds`，UI 层却设置了屏障：默认的 Sentence Card 视图只有“调整高亮色（Color Swatches）”。用户必须显式点击 “Edit tags / 添加标签”（变更 `showTagEditor = true` 显式控制）后，原有的全局标签列表 `highlightStore.tags` 才会由于 `v-if` 的条件被展开挂载出来供用户挑选 (`toggleExistingTag`)。

### 9.4 根因候选
1. **过度收纳的 UI 状态门（UX割裂）**：操作区强制分类，将“挂原有标签”和“加新标签”视为了非常规高阶行为而折叠隐藏。
2. **缺乏新建与绑定的一体化联结流**：没有在创建高亮的同屏即时展示最热 Top N 个已有标签，造成二次交互流失。

### 9.5 双端差异
- **PC端**：屏幕够大，二次点开增加的内高可接受但操作啰嗦。
- **移动端**：对寸土寸金的空间尤甚，且点击动作繁琐，打开二级折行易诱发高度裁掉的新问题。

### 9.6 修复建议
- **展平操作**：废除 `showTagEditor` 这个强制关卡；对于高频使用场景，高亮操作栏应当默认“前置推荐并展示已有标签”，使已有 `tag` 和颜色拾取器并列呈现（采用水平排布的 chip-row 形式）。对超过数量限制的，再采取 `More` / 滑动呼出折叠态。

## 10. 总结与改进优先级

### 共性根因归总
1. **缺乏现代布局适配机制（定位与事件的丢失）**：缺乏 Observer 或滚动的跟踪补偿（卡片裁切）；全局 PointerDown 漏绑（无法关闭）。
2. **DOM特性的误用**：违背 CSS 内联高亮模型的 `<button>` 强包裹导致段落高亮畸形。
3. **架构的扁平化重压**：无论侧栏目录还是标签交互，均因拒绝引入 Tabs 或智能分页组引起纵向信息塌陷。

### 下一轮开发优先级建议
**P0 严重级阻断项（必须第一顺位处理）：**
- **无法闭合外层卡片**：因缺字不触发。属于秒修复秒收益项，加上 listener 即可。
- **高亮区域错误**：极其影响阅读观感与业务交付心智。改 `span` 重构模型优于带病延展。
- **假翻译联调对接**：没有真数据功能就不算完备，不接受 "Unavailable" 作为产品的妥协。

**P1 功能体验瓶颈（强烈建议调优）：**
- **句子卡片定位与裁切问题**：需从 `fixed clientY` 重构为真实的 Floating Reference 追踪和 viewport 校准。
- **句子处老标签一键复用**：需要解禁 UI 门槛。

**P2 结构层沉淀项（后期规划）：**
- **Structure 面板信息层级改造**：通过 Tabs 或双段独立伸缩解决 PC 端拉伸覆盖，梳理移动端嵌套 Drawer 的合理高度。
